# docs/ 目录索引

> **本文件作用**：AI 首次进入 `docs/` 目录时，优先阅读此文件，快速理解整个文档体系的结构和每个位置的用途，避免在错误的文件中查找信息。

---

## 目录结构速查

```
docs/
├── README.md                          ← 你在这里（本文件）
├── 00-项目全景.md                     【项目总览】产品定位、技术栈、核心流程、目录结构
├── 01-架构决策记录.md                 【技术决策】DEC-001 ~ DEC-xxx，所有架构决策的权威记录
├── 02-文档更新规则.md                 【治理规则】什么时候必须更新文档、更新流程、责任归属
├── 03-任务边界约束.md                 【执行模板】每次具体任务前的约束清单（可复制使用）
├── analyses/                          【分析报告】深度分析、战略思考、外部对标
├── decisions/                         【专项决策】跨领域重大决策（商业、开源、品牌）
└── tasks/                             【任务记录】每一次具体开发/设计任务的执行记录与复盘
```

## 各文件/文件夹详细说明

### 根目录文件（按编号顺序阅读）

| 文件 | 作用 | 什么时候读 |
|------|------|-----------|
| `00-项目全景.md` | 项目的"身份证"。包含产品定位、技术栈、核心流程、目录结构、快速启动命令。 | **每次新会话首先读** |
| `01-架构决策记录.md` | 技术架构的"宪法"。记录为什么选 FastAPI、为什么用 SSE、为什么六阶段模型等。含选项对比和决策理由。 | 讨论技术方案时读 |
| `02-文档更新规则.md` | 文档治理的"操作规程"。明确什么代码改动必须同步更新文档、提交规范、发版前检查清单。 | 修改用户可见功能时读 |
| `03-任务边界约束.md` | 单次任务的"边界说明书"。可复制此模板，填入本次任务的具体约束（允许改哪些文件、禁止改哪些、验收标准）。 | 执行具体开发任务前读 |

### 子目录

#### `analyses/` — 分析报告
存放**非执行性**的深度思考内容：
- 项目全景深度分析（技术、产品、市场维度）
- 创始人战略关注清单
- 竞品分析、用户调研报告
- AI 适配度评估、国际化流程对照

**特点**：偏"战略层"，不直接指导代码编写，但指导方向选择。

#### `decisions/` — 专项决策
存放**跨领域重大决策**，通常涉及多个模块或长期影响：
- 开源战略与商业化模式
- 品牌定位调整
- 技术栈迁移决策
- 重大架构重构决策

**特点**：一个决策可能影响未来 3~6 个月的技术路线，需要单独归档。

#### `tasks/` — 任务记录
存放**每一次具体执行**的完整上下文：
- 任务目标、涉及模块、预计耗时
- 执行过程中的关键决策和遇到的问题
- 代码变更摘要
- 复盘（哪些约束有效、哪些应该调整）

**特点**：偏"战术层"，是 AI 学习项目历史、避免重复踩坑的核心素材。

## 使用规范

1. **新增文件时**：先判断内容类型，放入对应子目录，不要全堆在根目录。
2. **命名规范**：
   - 根目录编号文件：`NN-描述.md`（如 `01-架构决策记录.md`）
   - 子目录文件：`YYYY-MM-DD-描述.md`（如 `2026-04-27-邮箱注册与OAuth登录系统.md`）
3. **交叉引用**：文档之间提到彼此时，使用相对路径链接。例如：
   > 本任务涉及 OAuth 登录，相关架构决策见 `../01-架构决策记录.md` DEC-016。

---

> **维护者**：项目核心维护者。每次新增子目录或调整根目录文件时，同步更新本索引。

## 给 AI 的读取提示（准确性优先）

> **原则**：本项目 API 计费无限制，Token 不是瓶颈。**准确性才是瓶颈**。执行具体任务前，如阅读相关文档能提升回答质量，应主动读取，不要因为费 Token 而跳过。

**读取优先级**：

| 场景 | 必须阅读的文档 | 为什么 |
|------|---------------|--------|
| 新会话首次交流 | `00-项目全景.md` | 快速建立项目全局认知 |
| 讨论技术方案 / 做架构选择 | `01-架构决策记录.md` | 避免推翻已确定的决策，了解历史选型理由 |
| 执行具体开发任务 | `03-任务边界约束.md` + `tasks/` 下相关历史任务 | 了解本次任务的边界和前人踩过的坑 |
| 修改用户可见功能 | `02-文档更新规则.md` | 确保代码改动同步更新对外文档 |
| 制定产品方向 / 优先级 | `analyses/` + `decisions/` | 基于已有分析和决策做判断，不凭空想象 |

**特别提醒**：
- `tasks/` 目录下的历史任务记录包含大量执行细节和复盘，是避免重复踩坑的宝贵素材。涉及相似模块时，优先检索相关任务记录。
- `01-架构决策记录.md` 中的 DEC 条目具有约束力。如果用户的请求与已有 DEC 冲突，应主动指出并讨论是否需要推翻，而不是默默忽视。
