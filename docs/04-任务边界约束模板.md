# 任务边界约束模板（Level 3 上下文）

> **文档用途**：每次执行具体任务时，复制本模板并填入本次任务的临时约束（Level 3 上下文），作为 AI 执行时的边界条件。
>
> **使用方式**：
> 1. 复制本模板内容到对话中
> 2. 根据本次任务填写"本次任务信息"和"约束条件"
> 3. 作为 Prompt 的一部分发给 AI
>
> **原则**：约束要具体、可执行，不要写"不要出 bug"这种空话。

## 二、约束条件（必须遵守）

### 2.1 文件边界（明确哪些文件可以改，哪些不能改）

**允许修改的文件**：
- `frontend/src/components/xxx.tsx`
- `frontend/src/services/xxx.ts`
- （列出具体文件路径）

**禁止修改的文件**：
- `backend/alembic/versions/` 下的任何文件
- `frontend/src/types/index.ts`（除非本次任务明确涉及类型变更）
- `backend/app/models.py`（除非本次任务明确涉及数据库模型变更）
- （列出不可触碰的核心文件）

### 2.2 代码规范约束

- [ ] 使用命名导出（`export function`），禁止默认导出
- [ ] 类型定义优先 `interface` 而非 `type`
- [ ] 错误处理使用 `Result<T, E>` 模式（如项目已有此模式）
- [ ] 保持现有代码风格（缩进、命名、注释习惯）

### 2.3 功能约束

- [ ] 不要破坏现有功能（向后兼容）
- [ ] 不要删除已有的 API 端点（可标记 deprecated）
- [ ] 不要修改已有数据库表结构（如需修改，新增 migration）
- [ ] 前端修改后必须能通过 `npm run build`
- [ ] 后端修改后必须能正常启动（`python start.py`）

### 2.4 测试约束

- [ ] 修改后自动运行 Playwright E2E 测试（如已有相关测试用例）
- [ ] 如测试失败，分析错误并修复，不要直接跳过测试
- [ ] 新增功能建议补充对应的 E2E 测试用例

### 2.5 提交约束

- [ ] 完成后生成变更摘要（改了哪些文件、为什么）
- [ ] git commit 信息要清晰描述变更内容
- [ ] 如任务未完成（需要人决策），明确标注"阻塞点"和"待决策事项"
