# 开源战略与商业化模式决策

**日期**: 2026-04-27
**决策状态**: 已采纳
**影响范围**: 项目整体架构、代码仓库可见性、商业模式、长期维护策略

---

## 背景

SparkBin 目前是一个闭源开发的 Indie Hacker 项目管理工具，采用 FastAPI + React 技术栈。项目已具备完整的核心功能（6 阶段工作流、AI 代理、Stripe 支付基础设施、用户认证），但尚未面向真实用户发布。

在商业化路径的讨论中，我们面临一个根本性选择：
- **闭源收费**: 直接以 SaaS 形式售卖，功能即壁垒
- **开源获客 + Cloud 托管收费**: 代码开源建立信任，托管版本创造收入

---

## 考虑过的选项

### 选项 A：纯闭源收费 SaaS

**优势**：
- 技术壁垒即商业壁垒，竞争对手难以复制
- 代码不暴露，安全性审计压力小
- 可直接按功能订阅收费

**劣势**：
- 作为独立开发者，无法与 Notion/Linear 等大厂在营销预算上竞争
- Vibe Coder / Indie Hacker 群体对闭源工具的接受度持续下降
- 缺乏社区贡献和免费传播渠道
- 项目依赖单一维护者（作者本人），闭源模式下 burnout 风险极高

### 选项 B：纯开源，无托管收费

**优势**：
- 社区传播效应最强，容易获得 GitHub Stars 和 HN 曝光
- 外部开发者可贡献代码，减轻维护负担

**劣势**：
- 没有收入来源，1000 元/月的 API 成本成为纯支出
- 99% 的 Vibe Coder 不会自己部署后端，实际用户量极低
- 没有收入意味着没有长期维护动力，项目极易 abandoned

### 选项 C：开源 + Cloud 托管收费（已采纳）

**核心逻辑**：
- **开源解决信任问题**：用户可以看到代码如何处理自己的商业机密（项目想法、痛点分析、定价策略）
- **开源解决传播问题**：GitHub Stars、HN Show、Reddit r/SideProject、Twitter/X 都是免费的获客渠道
- **托管收费解决可持续性问题**：为有付费意愿但不愿自己部署的用户提供便利，覆盖 API 和服务器成本
- **自托管选项解决锁定问题**：用户始终可以选择自己部署，不会被平台绑架

**定价设计**：

| 档位 | 月付 | 年付 | 项目数 | AI 调用/月 | 核心差异 |
|------|------|------|--------|-----------|---------|
| Free | $0 | $0 | 3 个 | 30 次 | 体验完整流程 |
| Pro | $9 | $79 | 无限 | 500 次 | 深度分析、导出 PDF、GitHub 自动备份 |
| Team | $29 | $249 | 无限 | 2000 次 | 实时协作、团队看板 |

**成本测算**（基于 DeepSeek API）：
- 单次 AI 调用约 2000 token，成本 ¥0.004
- 50 个 Pro 用户 × 500 次 = ¥100/月 API 成本
- 收入 50 × $9 = $450 ≈ ¥3200/月
- 1000 元预算可支撑约 400 个 Pro 用户的 AI 消耗

---

## 决策

**采用选项 C：开源（MIT License）+ Cloud 托管收费模式。**

---

## 影响

### 对代码结构的影响

1. **敏感配置必须完全环境变量化**
   - Stripe 密钥、JWT Secret、数据库 URL、SMTP 密码等已环境变量化
   - 需要确认 `.gitignore` 已正确排除 `.env` 和数据库文件
   - 需要创建 `.env.example` 作为配置模板

2. **登录系统安全性要求提高**
   - 代码公开后，攻击者可以审计认证逻辑
   - 当前已实现的 Rate Limit、bcrypt 密码哈希、JWT 过期机制是必要基础
   - 未来添加的 OAuth（GitHub/Google）也需确保 client_secret 不外泄

3. **自部署文档成为刚需**
   - 需要 `SELF_HOSTING.md` 说明 Docker/裸机部署步骤
   - 需要说明最低系统要求（Python 3.11+、SQLite/PostgreSQL）

### 对商业模式的影响

1. **收入与用户成功绑定**：用户用 SparkBin 做出的产品越成功，付费意愿越强
2. **社区即护城河**：开源社区的贡献和口碑比功能壁垒更持久
3. **Cloud 版是 convenience 收费**：卖的是"省时间"，不是"功能"

### 对开发节奏的影响

1. **所有代码默认公开**：需要更高的代码质量和注释标准
2. **Issue 和 PR 管理**：需要建立贡献者指南（CONTRIBUTING.md）
3. **版本发布策略**：需要语义化版本（SemVer）和 changelog
