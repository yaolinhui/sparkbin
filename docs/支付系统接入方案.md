# SparkBin 支付系统接入方案

> 目标：支持"开源+SaaS 双轨"模式，自托管用户免费使用，SaaS 用户按需付费。

## 一、额度策略评估与建议

### 1.1 当前配置

```python
# backend/app/config.py
credits_grant_on_register = 20          # 注册赠送额度
credits_packs = "5:100,10:250,20:600"   # 价格(USD):额度数
```

| 额度包 | 价格 | 单价 |
|---|---|---|
| 100 Credits | $5 | $0.05/次 |
| 250 Credits | $10 | $0.04/次 |
| 600 Credits | $20 | $0.033/次 |

### 1.2 20 次免费额度评估

**结论：偏低。**

一个完整的项目通常需要 4 个阶段（想法→验证→原型→发布），每个阶段平均 10-15 次 AI 交互，总计约 50-80 次。20 次额度只能覆盖前 2 个阶段的粗略体验，用户尚未感受到完整价值就被截断，转化率会受影响。

**建议方案（三选一）：**

| 方案 | 免费额度 | 机制 | 适用场景 |
|---|---|---|---|
| A. 提升注册赠送 | **50 次** | 注册即送，用完即购 | 追求高转化率，产品体验闭环完整 |
| B. 每日签到奖励 | **20 次** + 每日登录送 3 次 | 鼓励日活，防止一次性薅完 | 追求用户留存和活跃度 |
| C. 项目制免费 | 每个项目送 20 次 | 按项目维度赠送 | 鼓励用户多创建项目，但需防刷 |

**推荐方案 A（50 次注册赠送）**：

- 50 次足够完成 1 个完整项目，用户能体验到"从想法到上线"的完整闭环
- 定价梯度合理：用完 50 次后，$5 买 100 次是"低门槛首购"，转化率高
- 实施简单：只改 `credits_grant_on_register = 50`

## 二、支付系统全景对比

### 2.1 海外支付

| 系统 | 费率 | 优势 | 劣势 | 适合谁 |
|---|---|---|---|---|
| **Stripe** | 2.9% + $0.30/笔 | 文档最完善，支持信用卡/Apple Pay/Google Pay，测试模式成熟，Webhook 可靠 | 需海外公司或个体工商户，国内收款需结汇 | **首选**，已有代码基础 |
| **PayPal** | 3.49% + 固定费 | 用户认知度高，无需商户号即可个人收款 | 体验老旧，退款争议多，提现手续费高 | 备选，个人开发者快速验证 |
| **LemonSqueezy** | 5% + $0.50/笔 | 自动处理全球税务（VAT/GST），适合独立开发者，支持订阅和一次性付款 | 手续费高，审核严格，中国大陆地区支持有限 | 不想处理税务的独立开发者 |
| **Paddle** | 5% + $0.50/笔 | 类似 LemonSqueezy，支持订阅管理 | 审核周期长，中国大陆支持有限 | 计划做订阅制的 SaaS |

### 2.2 国内支付

| 系统 | 费率 | 资质要求 | 优势 | 劣势 |
|---|---|---|---|---|
| **微信支付** | 0.6% | 企业/个体户 + 对公账户 | 覆盖最广，用户习惯最强 | 个人开发者无法直接接入 |
| **支付宝** | 0.6% | 企业/个体户 + 对公账户 | 技术文档完善，支持花呗分期 | 同微信，需资质 |
| **云闪付** | 0.38%-0.6% | 企业/个体户 | 费率略低 | 用户量少 |

**国内个人开发者绕行方案：**

- 通过 **Stripe** 收人民币（Stripe 支持 CNY 结算，但需香港/海外公司）
- 使用 **LemonSqueezy**（支持支付宝/微信支付作为付款方式，但需通过其平台）
- 使用第三方聚合支付（如有赞、微盟），但手续费更高（通常 1%-2%）

## 三、SparkBin 当前支付架构

### 3.1 已有能力

```
前端
├── UpgradePromptModal.tsx      # 额度用完弹窗
├── PricingPreview.tsx          # 项目定价预览（Stripe Test Mode）
├── PaymentResultModal.tsx      # 支付结果页
└── paymentsApi (services/api.ts)  # 支付接口封装

后端
├── routers/payments.py         # Stripe Checkout + Webhook
├── models.py                   # User.ai_credits / CreditTransaction
├── config.py                   # enable_payments / credits_packs / stripe_keys
└── schemas.py                  # 支付相关 DTO
```

### 3.2 核心流程（已实现）

```
1. 用户点击"购买额度"
   → 前端调用 POST /payments/purchase-credits
   → 后端创建 Stripe Checkout Session
   → 用户跳转到 Stripe 支付页

2. 支付成功
   → Stripe 回调 POST /payments/webhook
   → 后端验证签名，更新 user.ai_credits
   → 写入 CreditTransaction 流水

3. 用户回到前端
   → /payment/success 页面
   → 前端调用 GET /payments/credits-status 刷新额度
```

### 3.3 关键配置

```ini
# backend/.env
ENABLE_PAYMENTS=true
CREDITS_GRANT_ON_REGISTER=50          # 建议从 20 改为 50
CREDITS_PACKS=5:100,10:250,20:600
STRIPE_SECRET_KEY=sk_test_xxx
STRIPE_PUBLISHABLE_KEY=pk_test_xxx
STRIPE_WEBHOOK_SECRET=whsec_xxx
```

## 四、接入方案详解

### 4.1 海外 SaaS（推荐：Stripe）

**你已经有 80% 的代码了**，只需补充以下环节：

#### Step 1: 注册 Stripe 账户

1. 访问 [stripe.com](https://stripe.com)，用邮箱注册
2. 切换到 **Test Mode**（无需真实资质即可开发测试）
3. 获取 API Keys：`Publishable key` + `Secret key`

#### Step 2: 配置 Webhook

1. Stripe Dashboard → Developers → Webhooks
2. Add endpoint: `https://your-domain.com/payments/webhook`
3. 选择事件：`checkout.session.completed`
4. 复制 `Signing secret` 到 `.env` 的 `STRIPE_WEBHOOK_SECRET`

#### Step 3: 本地测试 Webhook

```bash
# 安装 Stripe CLI
stripe login
stripe listen --forward-to http://localhost:8000/payments/webhook
```

#### Step 4: 测试支付流程

1. 前端点击购买 → 跳转到 Stripe Checkout
2. 使用 Stripe 测试卡号：`4242 4242 4242 4242`
3. 任意未来日期 + 任意 CVC + 任意邮编
4. 支付成功 → Webhook 触发 → 后端充值额度

#### Step 5: 上线切换

1. Stripe Dashboard → 切换为 Live Mode
2. 更新 `.env` 为 Live API Keys
3. 重新配置 Webhook endpoint 为生产域名

### 4.2 国内 SaaS（微信支付 + 支付宝）

**资质门槛：**

- 企业营业执照 或 个体工商户执照
- 对公银行账户（个体工商户可用法人个人银行卡）
- ICP 备案（域名需备案）

**接入方式：**

| 系统 | 接入路径 | 开发复杂度 |
|---|---|---|
| 微信支付 | [微信支付商户平台](https://pay.weixin.qq.com/) → 申请商户号 → 接入 JSAPI / H5 支付 | 高（需签名、证书、回调验签） |
| 支付宝 | [支付宝开放平台](https://open.alipay.com/) → 申请商户号 → 接入手机网站支付 | 中（SDK 较完善） |

**建议：**

- 如果目标用户在国内且你有资质，优先接入**微信支付 JSAPI**（体验最好）
- 如果暂时没有资质，先用 **Stripe** 收海外用户，国内用户引导至"自托管部署"

### 4.3 自托管模式（开源用户）

自托管用户应完全关闭支付功能：

```ini
# backend/.env
ENABLE_PAYMENTS=false
CREDITS_GRANT_ON_REGISTER=999999    # 或极大值
```

前端行为：
- `UpgradePromptModal` 检测到 `packs.length === 0` 时显示"当前为自托管模式，无需购买额度"
- AI 调用时不检查额度（或额度永远用不完）

## 五、定价策略建议

### 5.1 海外定价（USD）

基于当前配置优化：

| 档位 | 价格 | 额度 | 单价 | 定位 |
|---|---|---|---|---|
| 免费试用 | $0 | 50 次 | - | 体验完整项目流程 |
| 入门包 | $5 | 120 次 | $0.042 | 低门槛首购，适合个人项目 |
| 进阶包 | $12 | 350 次 | $0.034 | 性价比最高，主推档位 |
| 专业包 | $29 | 1000 次 | $0.029 | 重度用户，团队使用 |

**为什么调整？**
- 原 "$5/100次" 单价 $0.05，在 AI 工具中偏高
- 增加 "$12/350次" 作为**主推档位**（锚定效应）
- 增加 "$29/1000次" 覆盖重度用户

### 5.2 国内定价（CNY）

如果接入国内支付，建议单独定价：

| 档位 | 价格 | 额度 | 说明 |
|---|---|---|---|
| 免费试用 | ¥0 | 50 次 | 同上 |
| 入门包 | ¥18 | 120 次 | 约 $2.5，符合国内用户心理价位 |
| 进阶包 | ¥48 | 350 次 | 约 $6.6，主推 |
| 专业包 | ¥128 | 1000 次 | 约 $17.6 |

### 5.3 订阅制 vs 一次性

**当前方案：一次性购买额度（永久有效）**

- 优势：用户无压力，购买即拥有，适合低频工具
- 劣势：收入不可预测，缺乏持续现金流

**可选演进：订阅制**

| 模式 | 月付 | 额度/月 | 适合 |
|---|---|---|---|
| 轻量版 | $4.9 | 200 次/月 | 个人开发者 |
| 专业版 | $12.9 | 无限额度 | 重度用户 |
| 团队版 | $29.9 | 无限额度 + 5 人 | 小团队 |

**建议：**
- **现阶段保持一次性购买**：开发成本低，用户决策门槛低
- **MRR 稳定后引入订阅制**：作为 Pro 版，额外解锁高级功能（如更多 AI 模型选择、团队协作）
