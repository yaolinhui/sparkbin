# 路线 B：流式逐维度输出 —— 用户体验深度分析

> 分析日期：2026-04-30
> 分析对象：IdeaStage "AI建议" 按钮的生成体验优化
> 核心问题：单请求 17-25s 空白等待，用户焦虑严重

---

## 一、路线 B 的定义

**不是简单的"加个 loading 动画"，而是：**

1. **后端协议改变**：`POST /ai/idea-suggest` 从"请求-响应"模式改为 **SSE 流式推送**
2. **Prompt 格式改变**：从"生成完整 JSON"改为"逐行按维度输出"，例如：
   ```
   核心痛点：想法太多太散，缺乏系统化管理工具
   目标用户：独立开发者与 indie hacker
   使用场景：记录灵感、验证需求、追踪开发进度
   解决方案：阶段化看板，从想法到发布
   差异化价值：专为独立开发者设计，极简快速
   ```
3. **前端解析改变**：EventSource 实时接收，每完成一行立即填充对应卡片
4. **视觉呈现改变**：模态框打开瞬间显示 5 个空便利贴，内容像打字机一样逐个出现

## 二、当前系统架构兼容性分析

### 2.1 后端能力评估

**已有的 SSE 基础设施（✅ 可复用）：**
- `/ai/chat` 端点已经使用 `StreamingResponse` + SSE 格式
- `AIProxyService.chat_completion()` 已经支持 `stream=True`
- `ai_proxy.py` 中的 chunk 解析逻辑可以复用

**需要新增/修改的部分：**
- `generate_idea_suggestions()` 当前使用 `stream=False`，需要新增流式版本
- 需要实现"增量行解析器"——在 SSE chunk 流中识别完整的维度行
- 需要保证不同 provider（DeepSeek/Kimi/Doubao）的 chunk 分割行为一致

**技术风险：高。**

不同 provider 的流式行为差异：
- DeepSeek：chunk 较小，逐字/逐词推送，适合实时打字机效果
- Kimi：chunk 较大，有时整句推送，可能出现"卡顿后突现一行"
- Doubao：chunk 大小不稳定

这意味着**同一段代码在不同 provider 下的用户体验不一致**——今天在 DeepSeek 下是丝滑的打字机，明天 fallback 到 Kimi 就变成"卡住3秒然后蹦出一行"。

### 2.2 前端能力评估

**已有的流式消费基础设施（✅ 可复用）：**
- `ai.ts` 中的 `chatCompletionStream()` 已经实现了 SSE 解析器
- `AIChat.tsx` 已经实现了流式文本的打字机渲染

**需要新增/修改的部分：**
- `IdeaSuggestModal` 需要从"等完整数据后渲染"改为"增量渲染"
- 需要管理"部分接收状态"——比如收到了"核心痛点"的内容，但还没收到"目标用户"
- 需要处理流中断的降级逻辑（只收到3个维度，剩下2个怎么办？）

## 三、用户体验的逐层拆解

### 3.1 时间感知层（Perceived Time）

**心理学事实：**
- 空白等待时，用户感知时间 ≈ 实际时间 × 1.5~2.0
- 有进度反馈时，用户感知时间 ≈ 实际时间 × 0.6~0.8
- 有内容出现时，用户感知时间 ≈ 实际时间 × 0.4~0.6

**路线 B 的效果预估：**
- 实际时间：18s（不变）
- 感知时间：约 10-12s（因为有内容逐个出现）
- **感知提速：约 40%**

**但有一个陷阱：**
如果第一个维度（核心痛点）在 8 秒后才出现，用户前 8 秒仍然是空白等待，焦虑感不会减少。真正关键的是**首维度出现时间（First Dimension Time, FDT）**。

### 3.2 认知负荷层（Cognitive Load）

**当前模式（一次性出现）：**
- 用户等待 18 秒后，5 个卡片同时出现
- 用户可以快速扫视全部内容，判断整体质量
- 如果某一条很差，用户一眼就能看到
- **认知模式：整体扫描 → 局部深入**

**路线 B 模式（逐个出现）：**
- 第 1 秒：出现"核心痛点"，用户开始阅读
- 第 5 秒：出现"目标用户"，用户注意力被吸引过去
- 第 10 秒：出现"使用场景"，用户还没读完前两条
- **认知模式：持续被新信息打断，难以形成整体判断**

**对 indie hacker 用户的特殊影响：**
- 这类用户通常是"扫描型阅读者"，习惯快速浏览再决定深入哪里
- 逐个出现迫使他们变成"顺序阅读者"，效率降低
- **类比**：就像看 Netflix 时，一集一集自动播放 vs. 一次性放出全季

### 3.3 控制感层（Sense of Control）

**路线 B 的潜在问题：不可打断的流**

一旦 SSE 连接建立，用户看到内容在流式出现，此时：
- 点"取消"？可能前端已经显示了部分卡片，取消后状态混乱
- 发现第一条就很离谱？必须等全部流完才能关闭，否则可能只保存了部分建议
- 想切换"智能合并/全部覆盖"模式？流式过程中模式选择没有意义

**当前模式的优势：**
- 等待期间只有一个"取消"按钮，状态简单
- 结果一次性出来，用户做一次性决策

## 四、实现复杂度的真实评估

### 4.1 后端改动量

| 模块 | 改动点 | 复杂度 |
|------|--------|--------|
| `ai_proxy.py` | 新增 `generate_idea_suggestions_stream()` | 中 — 复用现有流式逻辑 |
| Prompt 工程 | 设计"流式友好"的 prompt，要求逐行输出 | **高** — 需要大量测试确保格式稳定 |
| 增量解析器 | 在 SSE chunk 中识别完整维度行 | **高** — chunk 可能被截断在行中间 |
| `ai.py` 路由 | 新增 `POST /ai/idea-suggest-stream` | 低 |
| Provider 兼容 | 测试 DeepSeek/Kimi/Doubao 的流式行为一致性 | **高** — 行为差异大 |

**Prompt 设计的具体难点：**

当前 prompt 要求返回严格 JSON，模型被"格式锁死"，输出稳定。如果改为自由文本逐行输出：

```
核心痛点：想法太多太散，缺乏系统化管理工具
```

模型可能输出：
```
好的，我来分析一下你的项目：

核心痛点：
你的想法目前比较分散，缺乏一个统一的管理工具来系统化地组织和追踪。

目标用户：
...
```

多了解释性文字、空行、markdown 格式——解析器需要处理所有这些噪声。

**Fallback 策略必须存在：**
如果流式解析失败（比如某一行没识别出来），必须能回退到"等全部内容收完再一次性解析"。这实际上要求维护两套解析逻辑。

### 4.2 前端改动量

| 模块 | 改动点 | 复杂度 |
|------|--------|--------|
| `api.ts` | 新增 `suggestIdeaNotesStream()` 返回 AsyncGenerator | 中 |
| `ai.ts` | 新增 `generateIdeaSuggestionStream()` 消费 SSE | 中 |
| `IdeaSuggestModal` | 从"等数据"改为"增量渲染"，管理部分状态 | **高** |
| 骨架屏 | 5 个空卡片 + 打字机动画 | 中 |
| 错误处理 | 流中断后的降级展示 | **高** |

## 五、路线 B 的隐藏风险

### 风险 1："半成品焦虑"

用户看到前 2 个维度很不错，但后 3 个维度迟迟不来（因为模型在"思考"后面的内容）。这种"半成品"状态比"全空白"更让人焦虑。

**类比**：餐厅上菜，前两道很快，后面三道等了 15 分钟——你会比"15 分钟后一次性上齐"更不耐烦。

### 风险 2：Provider 切换导致体验断裂

当 DeepSeek 不可用时，fallback 到 Kimi。但 Kimi 的流式 chunk 更大，用户会明显感觉到"卡顿"。这种不一致性会损害产品质感。

### 风险 3：编辑冲突

路线 B 的一个卖点是"用户可以先看先编辑"——但 `IdeaSuggestModal` 的设计是：等全部建议出来后才让用户选择"智能合并/全部覆盖"并编辑。如果允许用户在流式过程中就开始编辑第 1 条，而第 3 条还在生成，整个状态管理会变得极其复杂。

### 风险 4：缓存失效

当前有 1 小时的 LRU 缓存。如果改为流式，缓存的是完整结果还是流式过程？如果是完整结果，那缓存命中时仍然是"一次性出现"，体验不统一。

## 六、与当前用户画像的匹配度

### SparkBin 核心用户特征

- ** Indie hacker / 独立开发者**
- **时间碎片化**：可能在地铁上、咖啡馆里、睡前 10 分钟使用
- **目标导向**：打开 App 是为了"完成某个任务"，不是"体验 AI 的流式输出"
- **耐心阈值低**：如果 5 秒内没看到有价值的内容，就开始焦躁
- **结果驱动**：更在意"建议好不好用"，而不是"AI 生成过程酷不酷"

### 路线 B 的匹配度评估

| 维度 | 匹配度 | 说明 |
|------|--------|------|
| 时间碎片化 | ⚠️ 中 | 流式要求用户保持注意力，不适合被打断的场景 |
| 目标导向 | ⚠️ 中 | 用户想要的是"5 条建议"，不是"看 AI 写建议的过程" |
| 耐心阈值 | ✅ 高 | 首维度如果能 3 秒内出现，确实能减少焦虑 |
| 结果驱动 | ⚠️ 中 | 流式过程中用户提前看到部分内容，可能提前判断"不好"而关闭 |

## 八、我的专业判断

### 路线 B 不是当前最优解，原因如下：

1. **投入产出比失衡**：需要改动后端协议、Prompt 格式、前端状态管理、Provider 兼容测试，预估 2-3 天工作量。但核心收益只是"感知时间从 18s 降到 10s"，实际时间没变。

2. **稳定性风险高**：流式解析的 edge case 太多（chunk 截断、provider 行为差异、网络中断），维护成本高。对于一个"建议生成"这种低频功能，不值得承担这种技术债务。

3. **与当前 UX 设计冲突**：`IdeaSuggestModal` 的设计哲学是"对比旧内容和新建议，让用户做一次性决策"。流式逐个出现破坏了这个对比逻辑。

4. **有更便宜的方案能达到 80% 效果**：
   - **预生成**：用户点击时 80% 情况下已经准备好了
   - **骨架屏**：即使要等，5 个卡片骨架比"一只猫 + 三个点"更有进度感
   - **Prompt 精简**：当前 prompt 有 500+ 字，精简后可能减少 3-5s 实际时间

### 路线 B 的适用时机：

如果未来满足以下所有条件，再考虑实施路线 B：
- 用户量增长到需要"AI 体验差异化"作为卖点
- 有专门的前端工程师维护流式状态逻辑
- 确定了唯一的 provider（不需要处理多 provider 流式差异）
- 该功能的使用频率从"偶尔"提升到"高频"（比如每次编辑自动触发）

## 九、推荐实施路线：A + D 混合

**即：预生成骨架屏 + Prompt 精简优化**

具体：
1. **前端**：进入 IdeaStage 时自动调用 AI（后台静默请求）
2. **前端**：点击"AI建议"时，若已准备好则瞬间弹出；若未好则显示 5 个骨架卡片
3. **后端**：精简 `generate_idea_suggestions` 的 prompt，移除冗余约束，尝试降低 `max_tokens` 从 350 到 250
4. **后端**：测试 DeepSeek 是否比 Kimi 更快，考虑默认使用更快的模型

**预期效果：**
- 80% 的情况下：点击即得（0s 等待）
- 20% 的情况下：骨架屏 + 预计 12-15s 实际等待（比现在的 18-25s 快）
- 总工作量：半天到一天
- 稳定性：高（复用现有架构，不改协议）
