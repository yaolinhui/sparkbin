## 一、本次任务目标

将 Peter Steinberger 的"闭合循环（Closing the Loop）"方法论落地到 SparkBin 项目中：
1. **建立 Playwright E2E 测试体系**：覆盖核心用户流程（登录→项目列表→创建项目→阶段编辑）
2. **建立自动化验证流程**：AI 修改代码后自动运行构建 + E2E 测试，失败则自修，通过后才报告完成

4. **建立三层上下文体系**：Level 1（项目全景文档）✅ 已有 → Level 2（架构决策记录）→ Level 3（任务边界约束模板）

**最终目标**：人类开发者只需"发送需求 + 探讨架构"，之后的一切（编码、测试、修复、验证）由 AI 闭环完成，人类只需在本地运行并验收结果。

## 三、确定的技术方案

### 3.1 三层上下文文档体系

| 层级 | 文档 | 作用 | 更新频率 |
|------|------|------|---------|
| Level 1 | `SparkBin项目全景文档.md` | 项目全局认知 | 架构大变更时 |
| Level 2 | `002-架构决策记录.md` | 关键决策防遗忘 | 每次新决策时 |
| Level 3 | `003-任务边界约束模板.md` | 单次任务边界 | 每次任务前复制使用 |

### 3.2 Playwright E2E 测试方案

**安装配置**：
- 前端安装 `@playwright/test`（devDependencies）
- 配置 `playwright.config.ts`：自动启动前端 dev server + 后端服务
- 测试浏览器：Chromium（默认）

**测试用例范围（第一版）**：
1. `auth.spec.ts`：登录页面加载、登录成功/失败、Token 存储
2. `project-board.spec.ts`：项目列表加载、项目卡片渲染、Metrics Bar 筛选
3. `project-create.spec.ts`：创建项目弹窗、填写表单、提交成功
4. `project-detail.spec.ts`：项目详情页加载、StageFlow 导航、阶段切换
5. `theme.spec.ts`：主题切换（深色/浅色）、localStorage 持久化

**闭合循环集成**：
- AI 修改代码后自动执行：`npm run build` → `npx playwright test`
- 失败时分析错误日志，定位问题文件，自动修复
- 循环直到全部通过

### 3.3 Prompt 审查机制

- 创建 `prompts/` 目录
- 每次任务前保存 Prompt 到 `prompts/序号-时间-任务名.md`
- 任务完成后复盘："AI 哪里理解错了？Prompt 哪句话导致的？"

### 3.4 Git 分支隔离策略

**分支命名规范**：
```
feature/前端模块-功能简述    例：feature/login-loading-optimize
feature/后端模块-功能简述    例：feature/api-validation-enhance
fix/问题简述                例：fix/build-typescript-error
refactor/重构范围           例：refactor/stage-context-evaluate
```

**文件边界划分原则**：
- 前端组件修改 → `feature/frontend-xxx`
- 后端路由/服务修改 → `feature/backend-xxx`
- 数据库模型修改 → 单独分支，不与业务逻辑修改并行
- 公共类型修改 → 优先合并，其他分支 rebase

**合并策略**：
- "完成即合并"，不等其他 Agent
- 每小时检查一次分支状态
- 合并前必须构建通过 + 测试通过

### 3.5 任务颗粒度控制标准

**判断标准**：
| 指标 | 小任务（合适） | 大任务（需拆分） |
|------|---------------|-----------------|
| 预计耗时 | 30-60 分钟 | 超过 1 小时 |
| 涉及文件数 | 1-5 个 | 超过 10 个 |
| 跨越模块 | 单一模块（纯前端或纯后端） | 前后端同时修改 |
| 复杂度 | 修改已知逻辑 | 需要设计新架构 |

**拆分方法（以 SparkBin 为例）**：
- ❌ 大任务："给 ValidateStage 加上完整的跨列拖拽功能"
  - 涉及：DnD 逻辑重构、状态管理修改、后端 API 可能调整、测试用例编写
- ✅ 拆分后：
  - 任务 A："在 ValidateStage 前端实现跨列拖拽的 UI 交互（仅前端状态，不调 API）"
  - 任务 B："后端新增 updateValidationStatus API，支持批量更新验证项状态"
  - 任务 C："前端接入新 API，实现拖拽后的状态持久化"
  - 任务 D："补充 ValidateStage 跨列拖拽的 E2E 测试"

## 六、已知风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| Playwright 需要下载浏览器（约 100MB+） | 首次安装耗时 | 后台运行，不阻塞其他工作 |
| E2E 测试需要后端启动 | 测试环境依赖重 | 配置 `webServer` 自动启动前后端 |
| AI 自测自修可能陷入死循环 | 时间浪费 | 设置最大重试次数（如 3 次），超限后报告人决策 |
| 测试用例维护成本 | 后期负担 | 优先覆盖核心流程，避免过度测试 |
