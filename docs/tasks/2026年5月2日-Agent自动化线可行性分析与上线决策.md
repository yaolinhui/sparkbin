## 一、两条产品线的本质差异

| 维度 | 手动辅助线（当前） | 自动化 Agent 线（规划中） |
|------|-------------------|------------------------|
| **用户角色** | 主导者，AI 是参谋 | 监督者，AI 是执行者 |
| **交互方式** | 用户点击、编辑、确认 | 用户点击"启动 Agent"后全自动运行 |
| **AI 职责** | 生成建议内容（文本） | 自主决策、执行操作、推进阶段 |
| **技术复杂度** | 低：API 调用 + 展示 | **极高**：沙箱 + 状态机 + 错误恢复 |
| **出错影响** | 用户可随时修正 | Agent 可能把项目数据搞乱 |
| **完成度** | 80%+，已知 bug 可修 | 10%，核心架构未确定 |

## 二、自动化 Agent 线的技术可行性深度分析

### 2.1 核心难点拆解

#### 难点 A：沙箱执行环境（阻塞级）

**问题**：Agent 需要"运行代码并显示效果"，这意味着需要在服务器端执行用户生成的代码。

**技术方案对比**：

| 方案 | 实现难度 | 安全性 | 成本 | 可行性 |
|------|---------|--------|------|--------|
| Docker 容器隔离 | 高 | 高 | 中（需要 K8s 或容器编排）| 48h 内不可行 |
| Firecracker / gVisor | 极高 | 极高 | 高 | 超出 MVP 范围 |
| WebAssembly (WASM) | 中 | 高 | 低 | 可行但需重新架构 |
| 前端 iframe 沙箱 | 低 | 中（无法隔离恶意代码）| 极低 | 只能做演示，不能真正执行 |
| 云端 CodeSandbox API | 低 | 高 | 高（按量计费）| 最快实现，但依赖第三方 |

**结论**：沙箱执行不是一个"加个按钮"就能完成的功能，需要独立的基础设施层。

#### 难点 B：多阶段状态机与自动推进（高风险）

**问题**：Agent 如何判断"当前阶段已完成，可以进入下一阶段"？

**当前各阶段的完成标准**：

| 阶段 | 人工判断标准 | Agent 判断难度 |
|------|-------------|---------------|
| Idea | 5个便利贴都有非占位符内容 | 低（可正则匹配） |
| Prototype | 产品原型文档完整 | **高**（需要语义理解） |
| Validate | 验证结果有意义 | **极高**（需要业务逻辑判断） |
| Monetize | 定价策略合理 | **极高**（需要市场数据） |
| Grow | 增长计划可执行 | **极高**（需要外部数据） |
| Ship | 发布清单完成 | 中（可 checklist） |

**核心风险**：如果 Agent 误判阶段完成，会把用户带到下一个未准备好的阶段，导致数据混乱。这种错误不可逆。

#### 难点 C：Claude Code API 的现实约束（远期愿景）

**当前状态**：
- Claude Code 没有公开的 REST API
- Anthropic 提供的是 Messages API（纯文本对话），不是"代码生成 + 执行 + 文件操作"的 Agent API
- 要实现"生成代码、运行并显示效果"，需要自己搭建完整的 Agent 框架（ReAct / CoT / Tool Use）

**工作量估算**：
- 搭建基础 Agent 框架：2-3 周
- 集成代码生成与沙箱执行：2-3 周
- 多阶段状态机与错误恢复：2-3 周
- UI 实时反馈（SSE/WebSocket）：1 周
- **总计：2-3 个月全职开发**

### 2.2 与 48 小时上线目标的冲突

| 资源 | 手动辅助线上线需要 | Agent 线 MVP 需要 | 48h 内可用 |
|------|-------------------|------------------|-----------|
| 开发时间 | 12-16 小时 | 200+ 小时 | ✅ 只够前者 |
| 测试时间 | 4-8 小时 | 40+ 小时 | ✅ 只够前者 |
| 基础设施 | Render/Vercel 即可 | 需要沙箱集群 | ✅ 前者已就绪 |
| 风险窗口 | 可控（用户手动操作）| 高（自动操作可能破坏数据）| ✅ 前者安全 |

## 三、决策建议

### ✅ 强烈建议：48h 内只上线手动辅助线

**理由**：
1. **时间约束不可逆**：Agent 线需要 2-3 个月，48 小时连架构设计都做不完
2. **用户价值已足够**：手动辅助线的"AI 建议"功能已经能解决 indie hacker 的核心痛点（想不清楚产品方向）
3. **数据安全**：自动 Agent 有破坏用户数据的风险，手动线完全由用户控制
4. **快速验证**：先上线手动线，收集真实用户反馈，再决定 Agent 线是否值得投入

### 📋 Agent 线作为 V2 路线图（上线后 3-6 个月）

**阶段规划**：

```
V1（现在-48h后）：手动辅助线上线
  └── 注册/登录 → 项目管理 → 6阶段手动推进 → AI建议辅助

V2（1-2个月后）：增强手动辅助
  └── AI建议质量优化 → 更多维度 → 历史版本对比

V3（3-6个月后）：半自动 Agent（试点）
  └── 单个阶段内自动化（如：Idea阶段一键生成5个便利贴并自动保存）
  └── 用户确认后才推进到下一阶段

V4（6-12个月后）：全自动 Agent（远期）
  └── 沙箱执行环境搭建完成
  └── Claude Code API 或等效方案可用
  └── 多阶段全自动推进（高容错 + 用户随时可中断）
```

## 四、当前代码状态的紧急处理

### 问题：工作区混入了 Agent 相关开发代码

从 git 状态观察，工作区/其他分支存在大量未提交的组件和 hooks：
- `AgentCockpit.tsx`（Agent 驾驶舱）
- `PixelPetCompanion.tsx`（宠物伴侣）
- `AIPetASCII.ts`
- `GlobalToast.tsx` / `toastEvents.ts`

这些代码如果混入上线版本，可能导致：
1. 构建失败（依赖缺失）
2. 运行时错误（未完成的逻辑）
3. 包体积膨胀（无用代码）

### 处理方案

**方案 A（推荐）：功能门控隐藏**
- 保留代码在仓库中，但用 `enableAgentFeature = false` 门控隐藏入口
- 上线后不影响现有功能，后续开发继续在该代码上迭代
- 工作量：低（加几个 if 判断）

**方案 B：彻底剥离到独立分支**
- 把 Agent 相关代码从 deploy/production 移除，放到 `feature/agent` 分支
- 上线分支保持干净
- 工作量：中（需要处理文件移动和路径变更）

**方案 C：暂缓开发，代码留在工作区**
- 不上传到上线分支，但本地保留
- 风险：容易在后续合并时混入

## 六、督战结论

> **这不是一个"做不做 Agent"的问题，而是一个"在什么时间做"的问题。**

Agent 自动化线是正确的产品方向，但它是一个 **V2/V3 级别的功能**，不是 MVP 应该承载的。

当前 48 小时的核心目标是：**让海外 indie hacker 能注册、创建一个项目、在 Idea 阶段获得 AI 建议、并推进到 Ship 阶段。** 这个价值主张已经足够强，足够独立发布。

**建议立即行动**：
1. 确认"只上线手动辅助线"
2. 隐藏/剥离 Agent 相关未完成代码
3. 恢复部署工作（Zeabur + Vercel）
4. 48h 后收集反馈，再规划 Agent 线
