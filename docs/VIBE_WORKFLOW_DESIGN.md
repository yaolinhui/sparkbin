## 核心设计理念

**AI 生成 → 用户确认/修改**

不再是"左边编辑器 + 右边AI聊天"的刻板模式，而是每个阶段都有针对性的交互界面。

```
用户触发 → AI 生成内容 → 用户：✅ 满意 / ✏️ 修改 / 🔄 重试
```

## AI 宠物系统

### 概述
AI 宠物是贯穿整个产品的虚拟助手，用 ASCII 艺术呈现，增加陪伴感和趣味性，弱化工具属性。

### 宠物选项

| 宠物 | ASCII 形象 | 性格 | 语气风格 |
|------|-----------|------|---------|
| 🐱 猫咪 | `/\_/\`<br>`( o.o )`<br>`> ^ <` | 温柔鼓励型 | "这个想法很棒喵~ 让我帮你梳理一下..." |
| 🤖 机器人 | `[o_o]`<br>`/| |\`<br>`d b` | 高效理性型 | "分析完成。检测到3个核心痛点，建议优先级排序..." |
| 🐼 熊猫 | `(\/)/`<br>`(•ㅅ•)`<br>`/  っ` | 佛系陪伴型 | "慢慢来，不着急～我们先理清楚..." |
| 🦊 狐狸 | ` /^\`<br>`( ◠‿◠ )`<br>`/    \` | 犀利挑战型 | "这个想法有问题。用户真的会付费吗？" |

### 宠物属性

- **选择时机**：配置 AI 模型时同步选择
- **统一性**：选择后贯穿所有 6 个阶段
- **自定义命名**：用户可给宠物起名字
- **情绪调节**：设置里可拖动数值调节情绪风格（温柔/理性/佛系/犀利）
- **主动建议**：特定阶段会主动弹出给出建议

## 入口流程：初始化项目

### Step 1: 输入模糊想法

用户输入不清晰的想法：
```
┌─────────────────────────────────────────┐
│  // 初始化项目                           │
├─────────────────────────────────────────┤
│                                         │
│  输入描述                                │
│  ┌─────────────────────────────────────┐│
│  │ 我想做一个政策的信息化平台            ││
│  └─────────────────────────────────────┘│
│                                         │
│  [> AI 优化]                [跳过]      │
└─────────────────────────────────────────┘
```

### Step 2: AI 导师理解确认（强制）

点击"AI 优化"后，不直接生成标题，而是先确认理解：

```
┌─────────────────────────────────────────┐
│  // AI 导师理解确认                      │
├─────────────────────────────────────────┤
│                                         │
│         /\_/\                           │
│        ( o.o )   "让我理解一下..."       │
│         > ^ <                           │
│                                         │
│  您说的是：                               │
│  ┌─────────────────────────────────────┐│
│  │ "我想做一个政策的信息化平台"          ││
│  └─────────────────────────────────────┘│
│                                         │
│  我理解的核心问题（动态生成 2-5 个维度）：  │
│                                         │
│  ┌─ ① 信息分散 ─┐ ┌─ ② 更新滞后 ─┐     │
│  │   政策来源多   │ │   错过申报期   │     │
│  │   难以统一查看 │ │   造成损失     │     │
│  └─────────────┘ └─────────────┘     │
│                                         │
│  ┌─ ③ 风险预警 ─┐ ┌─ ④ 受众广泛 ─┐     │
│  │   容易违规     │ │   企业+个人都需要│    │
│  │   面临处罚     │ │   市场大        │    │
│  └─────────────┘ └─────────────┘     │
│                                         │
│        [✅ 理解正确，生成标题]            │
│        [✏️ 第③点不对，让我补充...]       │
│                                         │
└─────────────────────────────────────────┘
```

**递进修正逻辑**：
- 用户点击某个维度的"不对" → 展开输入框修改该点
- 其他维度保留不变
- 点击"重新理解"后仅更新该维度

**强制确认**：
- 用户必须点击"理解正确"才能进入下一步
- 不可跳过，不可关闭弹窗

### Step 3: 生成优化结果（确认参数）

```
┌─────────────────────────────────────────┐
│  // 确认参数                             │
├─────────────────────────────────────────┤
│                                         │
│  项目名称                                 │
│  ┌─────────────────────────────────────┐│
│  │ 政策智囊，实时预警，规避风险          ││
│  └─────────────────────────────────────┘│
│                                         │
│  描述                                     │
│  ┌─────────────────────────────────────┐│
│  │ 政策信息分散难寻、更新滞后，导致企业   ││
│  │ 或个人错失机遇、面临违规风险。         ││
│  └─────────────────────────────────────┘│
│                                         │
│        [< 返回]    [执行 >]              │
│                                         │
└─────────────────────────────────────────┘
```

用户可编辑生成的内容，确认后正式进入"想法"阶段。

## 通用交互模式

每个阶段都遵循统一流程：

```
AI 生成初稿 → 用户编辑/确认 → 推进到下一阶段
```

### 阶段衔接规则

1. **手动确认进入下一阶段** - 用户主动点击"完成此阶段"
2. **数据自动继承** - 上一阶段的关键内容带到下一阶段
3. **阶段回顾** - 进入新阶段时，AI 总结上一阶段产出

### AI 介入时机

- **进入阶段时** - 自动生成初稿
- **用户触发时** - 点击按钮生成特定内容
- **智能建议** - 根据用户输入实时补充

## 阶段分组：三大阶段

六个阶段按工作性质分为三大组，每组有明确产出：

```
┌─────────────────────────────────────────────────────────────┐
│  01 想法  →  02 验证  │  03 原型  →  04 发布  │  05 增长  →  06 变现  │
│                      │                      │                      │
│     【探索期】         │      【执行期】        │      【运营期】        │
│                      │                      │                      │
│   结论：可做/不可做     │   产出：可用产品       │   目标：持续收入       │
└─────────────────────────────────────────────────────────────┘
```

### 探索期（想法 + 验证）

**核心问题：** 这个想法值得做吗？

**产出：** GO / NO-GO 决策

| 结果 | 含义 | 后续动作 |
|------|------|---------|
| **GO (可做)** | 痛点真实、需求验证通过、有变现潜力 | 进入【执行期】开始开发 |
| **NO-GO (不可做)** | 痛点不存在、用户不愿付费、竞品太强 | 项目归档，记录原因，转向新想法 |
| **MAYBE (待观察)** | 需要更多信息或调整方向 | 回到"想法"阶段重新梳理 |

**关键设计：**
- 完成"验证"阶段后强制做 GO/NO-GO 决策
- 系统根据验证数据给出建议（如"5人访谈中4人表示愿意付费，建议继续"）
- NO-GO 不是失败，而是"快速排除"，节省后续时间

### 执行期（原型 + 发布）

**核心问题：** 如何最快做出可用的产品？

**产出：** 上线的产品（MVP）

**关键设计：**
- 强调"不完美也要发"，克服完美主义
- 发布检查清单确保基础可用
- 从探索期继承的核心功能优先做

### 运营期（增长 + 变现）

**核心问题：** 如何让产品持续赚钱？

**产出：** 稳定的收入流

**关键设计：**
- 从0到100个用户的内容策略
- 定价实验和转化优化
- 收入仪表盘实时反馈

### 交互规则

**1. 预生成便利贴**

初始化完成后，自动预生成 3-5 个便利贴：
- 01 痛点描述（继承初始化时的内容）
- 02 目标用户
- 03 场景故事
- 04 竞品对标
- 05 解决方案

每个便利贴内包含 AI 生成的初步内容，用户可直接编辑。

**2. 网格排列**

- 便利贴固定在网格中，每行最多 4 个
- 支持拖拽在同一行内调整顺序
- 不支持任意位置摆放，保持整齐

**3. AI 宠物互动**

- 宠物头像固定在右上角
- 自动在空白处弹出建议气泡：

```

      ( o.o ) → │ 检测到竞品分析不够 │
       > ^ <     │ 深入，要帮你补充吗？ │
                 │     [好] [暂时不用]  │
                 └────────────────────┘
```

**4. 完成阶段**

点击"完成此阶段"按钮后：
- 检查便利贴是否都有内容（至少 3 个填满）
- 弹出确认对话框：

```
┌─────────────────────────────────┐
│  确认完成"想法"阶段？              │
│                                 │
│  当前进度: 5/5 便利贴已填写 ✓      │
│                                 │
│  进入下一阶段"验证"后：            │
│  • 你可以继续编辑想法内容          │
│  • 需要验证"目标用户"和"解决方案"   │
│                                 │
│      [取消]    [进入验证阶段]      │
└─────────────────────────────────┘
```

### 数据流转

```
想法阶段产出:
├── 痛点描述 → 验证阶段：验证该痛点是否真实
├── 目标用户 → 验证阶段：找这类用户访谈
├── 场景故事 → 验证阶段：验证场景是否成立
├── 竞品对标 → 验证阶段：分析竞品弱点
└── 解决方案 → 验证阶段：验证方案是否被接受
```

## 详细设计：06 变现 (MONETIZE) 阶段

### 核心概念解释

变现（Monetization）= 让产品赚钱的方式。常见模式：

| 模式 | 说明 | 适合场景 |
|------|------|---------|
| **订阅制** | 月费/年费（如 $9/月） | 持续服务的工具类 |
| **一次性购买** | 买断制（如 $49） | 软件/课程 |
| **免费+付费墙** | 基础功能免费，高级功能收费 | 大部分 SaaS |
| **开源+捐赠** | 开源代码，靠打赏/赞助 | 开源项目 |
| **广告** | 展示广告获利 | 流量大的内容产品 |

核心指标：
- **MRR** = Monthly Recurring Revenue（月经常性收入）
- **ARPU** = Average Revenue Per User（每用户平均收入）
- **LTV** = Lifetime Value（用户终身价值）

## 增长与变现的路径选择

根据项目类型，用户可选择不同路径：

### 路径 A：先增长（适合社交/内容产品）
```
发布 → 增长（获取大量用户） → 变现（广告/增值服务）
```
- 例：小红书、Twitter 类内容产品

### 路径 B：先变现（适合工具/SaaS）
```
发布 → 变现（从第一天收费） → 增长（付费用户口碑传播）
```
- 例：Notion、Figma 类工具

### 路径 C：开源打赏（适合开源项目）
```
发布 → 增长（GitHub Star） → 变现（GitHub Sponsor/打赏）
```
- 例：开源库、开发者工具

**系统设计**：
- 在 05 增长阶段提示"选择你的变现路径"
- 根据选择调整 06 变现阶段的界面
- 支持路径切换（如先免费增长，后转为付费）

### 交互方式

**1. 点击节点**

- 滚轮缩放

**3. 连线含义**
- **实线** = 已存在的数据关联

- **绿色** = 已完成
- **黄色** = 进行中

### AI 辅助功能


**3. 路径预测**

```
因为目标用户（企业员工）更常在手机上查看政策
```

```
宠物配置

```

```
  │           │           │           │           │           │
```
