# 文档更新规则

> **文档用途**：明确 SparkBin 项目中文档的更新时机、更新范围和责任归属，防止文档与代码脱节。
>
> **适用范围**：所有面向用户和贡献者的文档（README、CONTRIBUTING、SELF_HOSTING、CHANGELOG、SECURITY 等）。
>
> **创建时间**：2026年4月27日

---

## 一、触发条件：什么时候必须更新文档？

满足以下**任意一条**，就必须同步更新相关文档。

| 触发条件 | 必须更新的文档 | 示例 |
|---------|--------------|------|
| 新增/修改了 `.env` 环境变量 | `SELF_HOSTING.md`、`.env.example` | 新增 `RESEND_API_KEY`，自托管文档必须说明配置方法 |
| 新增/删除了 API 端点 | `README.md`（功能列表） | 新增 `/auth/oauth/github` 回调 |
| 新增/删除了数据库字段或模型 | `CHANGELOG.md` | `users` 表新增 `email_verified` |
| 修改了启动方式或依赖 | `README.md`、`SELF_HOSTING.md`、`CONTRIBUTING.md` | 新增 `alembic upgrade head` 启动步骤 |
| 新增/移除了用户可见的功能 | `README.md`（Features）、`CHANGELOG.md` | 新增 OAuth 登录、功能门控 |
| 修改了代码规范 | `CONTRIBUTING.md` | 新增"禁止修改已有 migrations"规则 |
| 修复了安全漏洞 | `SECURITY.md`、`CHANGELOG.md` | Rate Limit 绕过修复 |
| 发布了新版本 | `CHANGELOG.md` | 每次发版前必须归档 Unreleased 内容 |

**核心判定标准**：

> 如果你的代码改动会导致一个**新用户**或**新贡献者**产生困惑，就必须同步更新文档。

---

## 二、反例：什么时候不用更新文档？

| 场景 | 为什么不用更新 |
|------|---------------|
| 纯 UI 样式调整（颜色、间距、字体大小） | 文档不描述像素级细节 |
| 内部重构（函数拆分、变量重命名，外部行为不变） | `CONTRIBUTING.md` 已覆盖命名规范 |
| Bug 修复且没有引入新用户行为 | 除非是安全相关，否则只写 `CHANGELOG.md` |
| 测试文件修改 | 不影响用户或贡献者 |
| 性能优化（无接口变化） | 可在 `CHANGELOG.md` 记一笔，不改主文档 |

## 三、更新流程

```
开发功能 → 本地测试 → 写/改代码
              ↓
    [代码改动是否影响用户或贡献者？]
              ↓
    是 → 同步更新相关文档 → 提交（含 docs: 标签）
    否 → 直接提交
              ↓
         提 PR → PR 模板勾选"文档已更新"
              ↓
         Reviewer 检查代码 + 文档一致性
              ↓
         合并
```

### 提交规范

凡是触发文档更新的代码改动，至少包含一个 `docs:` 提交：

```bash
git commit -m "docs: 更新 SELF_HOSTING.md，添加 OAuth 配置说明"
```

好处：
- `git log --grep="docs:"` 可快速定位文档相关提交
- CHANGELOG 生成工具可自动提取 `docs:` 提交

## 四、发版前文档冻结检查清单

每次发布新版本前，执行以下检查：

- [ ] `README.md` 功能列表与实际代码一致
- [ ] `SELF_HOSTING.md` 环境变量说明与 `.env.example` 一致
- [ ] `CHANGELOG.md` 已汇总本次版本所有改动
- [ ] 新增 API 端点已在 Swagger (`/docs`) 中可见
- [ ] `CONTRIBUTING.md` 代码规范未过期
- [ ] 安全相关改动已同步到 `SECURITY.md`

## 五、责任归属

| 文档 | 主要维护者 | 触发更新时谁负责 |
|------|-----------|-----------------|
| `README.md` | 核心维护者 | 功能改动提交者 |
| `SELF_HOSTING.md` | 核心维护者 | 部署/配置改动提交者 |
| `CONTRIBUTING.md` | 核心维护者 | 规范变更提出者 |
| `CHANGELOG.md` | 发版负责人 | 所有提交者（在各自 PR 中补充） |
| `SECURITY.md` | 安全负责人 | 安全修复提交者 |
