## 决策列表

### DEC-001：前后端分离架构

**日期**：2026-04-08

**决策内容**：将原本纯前端（localStorage）的 SparkBin 改造为前后端分离架构。

**选项对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| 保持纯前端 + localStorage | 简单，无部署成本 | 数据易丢失，无法多设备同步，AI 配置不安全 |
| 纯前端 + GitHub JSON 备份 | 有云端备份 | 需要暴露 GitHub Token，操作复杂 |
| **前后端分离（选中）** | 数据安全持久化，AI Key 存后端，可扩展用户系统 | 需要维护后端服务，部署成本增加 |

**决策理由**：
- AI 配置（API Key）必须安全存储，不能放在前端
- 需要用户认证和权限管理（admin / user）
- 未来可能需要多设备同步和协作功能

**影响范围**：全部项目架构

### DEC-002：后端技术栈选 FastAPI + SQLAlchemy

**日期**：2026-04-08

**决策内容**：后端使用 Python FastAPI 框架，ORM 使用 SQLAlchemy 2.0，数据库支持 SQLite（开发）和 PostgreSQL（生产）。

**选项对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| Node.js + Express + Prisma | 前后端同语言 | 对 AI 生成代码的友好度不如 Python |
| **FastAPI + SQLAlchemy（选中）** | 现代、类型安全、自动 API 文档、AI 生成质量高 | 需要维护 Python 环境 |
| Django | 功能全、生态成熟 | 过重，不适合轻量级 API 服务 |

**决策理由**：
- FastAPI 的 Pydantic 模型与前端 TypeScript 类型天然对应
- SQLAlchemy 2.0 支持类型提示，AI 不容易写错
- SQLite 适合本地开发，零配置启动

**影响范围**：`backend/` 目录全部代码

### DEC-003：AI 代理模式（后端代理）

**日期**：2026-04-09

**决策内容**：前端不直接调用任何 AI 提供商的 API，所有 AI 请求通过后端代理转发。

**选项对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| 前端直接调用 OpenAI/DeepSeek API | 简单，减少后端负载 | API Key 暴露在前端，安全风险极高 |
| **后端代理模式（选中）** | API Key 安全存储（加密），可统一限流/日志/切换模型 | 增加后端复杂度，需要维护多提供商适配 |

**决策理由**：
- API Key 安全性是底线，绝对不能暴露到前端
- 后端可以统一记录 AI 调用日志（AICallLog）
- 可以在后端实现模型切换、失败重试等逻辑

**影响范围**：`backend/app/services/ai_proxy.py`，`frontend/src/services/ai.ts`

### DEC-005：SSE 流式响应（而非 WebSocket）

**日期**：2026-04-10

**决策内容**：AI 聊天使用 SSE（Server-Sent Events）实现流式响应，而非 WebSocket。

**选项对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| WebSocket | 双向实时，适合高频交互 | 需要维护连接状态，前后端复杂度增加，防火墙/Nginx 配置麻烦 |
| **SSE（选中）** | 基于 HTTP，简单，自动重连，前端用 EventSource 即可 | 仅服务端向客户端推送，不适合双向高频场景 |

**决策理由**：
- AI 聊天是"一问一答"模式，SSE 完全够用
- SSE 基于标准 HTTP，调试简单（curl 可直接测试）
- 不需要维护 WebSocket 连接状态，减少 bug 来源

**影响范围**：`backend/app/routers/ai.py`（POST /chat），`frontend/src/services/ai.ts`

### DEC-006：六阶段 Vibe 工作流（IDEA → MONETIZE）

**日期**：2026-04-07

**决策内容**：项目生命周期固定为六个阶段：想法(IDEA) → 验证(VALIDATE) → 原型(PROTOTYPE) → 发布(SHIP) → 增长(GROW) → 变现(MONETIZE)。

**选项对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| 自由看板（任意列） | 灵活 | 缺乏引导，新手不知道该怎么走 |
| **固定六阶段（选中）** | 提供清晰的 Indie Hacker 路径，每个阶段有明确目标和退出标准 | 不够灵活，某些项目可能跳过某些阶段 |

**决策理由**：
- 产品定位就是"帮助 Vibe 开发者系统地推进项目"
- 固定阶段提供仪式感，降低决策疲劳
- 每个阶段可以设计专用的编辑器和 AI 辅助逻辑

**影响范围**：`frontend/src/types/index.ts`（StageKey），`backend/app/models.py`，六阶段组件

### DEC-009：前端状态管理使用 Zustand（而非 Redux）

**日期**：2026-04-08

**决策内容**：前端状态管理使用 Zustand，不使用 Redux 或 Context + useReducer。

**选项对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| Redux + RTK | 生态成熟，调试工具强 | 样板代码多，对 AI 不友好 |
| Context + useReducer | 无需额外依赖 | 性能差，组件树深时 re-render 严重 |
| **Zustand（选中）** | API 极简，无样板代码，TypeScript 友好，AI 容易写对 | 生态不如 Redux 丰富 |

**决策理由**：
- Zustand 的 API 极简，AI 生成的代码不容易出错
- 不需要 Provider 包裹，减少嵌套
- 中间件支持持久化、日志等

**影响范围**：`frontend/src/stores/projectStore.ts`，`frontend/src/stores/aiStore.ts`

### DEC-010：JWT Token 存储于 localStorage

**日期**：2026-04-08

**决策内容**：用户认证使用 JWT，Token 存储在浏览器 localStorage 中。

**选项对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| Cookie + HttpOnly | XSS 安全 | 需要处理 CSRF，跨域麻烦 |

**决策理由**：
- 简化实现，降低前后端复杂度
- 当前是单用户/小团队工具，安全要求不是最高优先级
- 如未来需要更高安全性，可迁移到 Cookie + Refresh Token 方案

**影响范围**：`frontend/src/services/api.ts`，`backend/app/auth.py`

### DEC-016：多用户认证体系（邮箱注册 + OAuth 2.0）

**日期**：2026-04-27

**决策内容**：将原有单一管理员认证系统升级为完整的多用户认证体系，支持邮箱注册（带邮件验证）和第三方 OAuth 登录（Google / GitHub）。

**选项对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| 保持单管理员模式 | 实现最简单 | 无法支持多用户，无法商业化 |
| 仅邮箱注册 | 完全自主可控 | 用户注册门槛高，转化率低 |
| **邮箱注册 + OAuth 2.0（选中）** | 兼顾自主注册和快捷登录，国际化友好 | 需要维护 OAuth 应用配置，依赖第三方服务可用性 |

**决策理由**：
- 产品定位面向全球 Indie Hacker，必须支持国际主流登录方式
- OAuth（Google/GitHub）大幅降低注册门槛，提升转化率
- 邮箱注册作为兜底方案，覆盖无法使用 OAuth 的用户
- 邮件验证机制防止垃圾注册
- Resend 邮件服务在开发环境可 mock，零成本启动

**技术细节**：
- 授权码流程（Authorization Code Flow）：安全，Token 不暴露给前端
- 回调后通过 URL query param 传递 token 到前端，前端 `OAuthHandler` 自动处理登录
- JWT 双 Token 体系保持不变（access 15min + refresh 7d）
- User 模型扩展 `oauth_provider` / `oauth_id` 字段区分来源
- `username` / `password_hash` 改为 nullable 以兼容 OAuth 用户

**影响范围**：`backend/app/routers/auth.py`, `backend/app/models.py`, `backend/app/email.py`, `frontend/src/components/LoginModal.tsx`, `frontend/src/App.tsx`, `frontend/src/pages/VerifyEmailPage.tsx`, `frontend/src/pages/ResetPasswordPage.tsx`

### DEC-017：SQLite 数据库文件排除版本控制

**日期**：2026-04-27

**决策内容**：将 `*.db` / `*.sqlite` / `*.sqlite3` 加入 `.gitignore`，并从 git 索引中移除已跟踪的 `backend/sparkbin_v2.db`。

**决策理由**：

- 不同开发者的本地数据库内容不一致，合并时会产生冲突
- 生产环境使用 PostgreSQL，SQLite 仅为开发用途

**影响范围**：`.gitignore`, `backend/sparkbin_v2.db`（停止跟踪，本地文件保留）

### DEC-018：产品方向转型——开源 + 社区会员模式

**日期**：2026-04-27

**决策内容**：将产品定位从纯商业 SaaS 转向**开源核心 + 社区会员增值**模式。核心功能完全开源免费，高级功能通过社区会员解锁。

**选项对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| 纯商业 SaaS（闭源） | 收入可控，代码保护 | 用户门槛高，社区无法参与，信任成本高 |
| 纯开源（无商业化） | 社区活跃，传播快 | 无持续收入，维护动力不足 |
| **开源核心 + 社区会员（选中）** | 社区驱动增长，核心功能透明可信，会员提供持续收入 | 需要维护两套功能边界，开源治理成本高 |

**决策理由**：
- 目标用户是 Indie Hacker，偏好可控、可审计、可自托管的工具
- 开源降低试用门槛，代码公开建立信任
- 社区会员（Pro/Team）为需要托管服务的用户提供便利，同时为项目提供可持续维护资金
- 自托管用户无限制（自带 API Key），与云服务形成差异化

**影响范围**：`README.md`（功能矩阵说明）、`LICENSE`（MIT）、`SELF_HOSTING.md`（部署指南）、功能门控系统（`backend/app/routers/` 中的权限检查）

### DEC-019：文档治理规则

**日期**：2026-04-27

**决策内容**：建立文档更新触发条件、提交规范（`docs:` 标签）、发版前文档冻结检查清单，防止文档与代码脱节。

**决策理由**：
- 项目快速迭代中，文档滞后是最常见的技术债务
- 需要可执行的规则而非依赖"有空再写"
- 开源项目文档是社区基础设施，文档质量直接等于项目专业度
- PR 模板中的"文档已更新"勾选项 + `docs:` 提交标签，形成最低成本的约束机制

**核心规则**：
1. 8 类必须更新文档的触发条件（新增 env 变量、API 端点、数据库字段、用户可见功能等）
2. 5 类不需要更新文档的场景（纯 UI 样式、内部重构等）
3. 发版前 5 项文档冻结检查清单
4. 文档债务跟踪表，定期 review

**影响范围**：`docs/02-文档更新规则.md`、`.github/PULL_REQUEST_TEMPLATE.md`、`CONTRIBUTING.md`

### DEC-020：全面安全加固（Security Hardening 2026-04-27）

**日期**：2026-04-27

**决策内容**：对 SparkBin 进行端到端安全审计并实施分层加固，覆盖认证、授权、输入验证、加密、日志脱敏、前端安全及传输安全。

**修复清单（按优先级）**：

| 优先级 | 领域 | 措施 | 文件 |
|--------|------|------|------|
| P0 | 启动安全 | 默认 SECRET_KEY / ENCRYPTION_KEY / admin/admin 强制拒绝启动，防止部署即裸奔 | `backend/app/config.py`, `backend/app/auth.py` |
| P0 | 数据隔离 | `/ai/call-logs` 增加 `user_id` 过滤，防止用户 A 查看用户 B 的调用记录 | `backend/app/routers/ai.py` |
| P0 | 信息泄露 | `/auth/me` 移除 `stripe_customer_id` / `stripe_subscription_id`，防止支付 ID 泄露 | `backend/app/routers/auth.py` |
| P0 | 信息泄露 | SSE 错误流仅返回 `"AI 服务暂时不可用，请稍后重试"`，隐藏内部异常类型和堆栈 | `backend/app/routers/ai.py` |
| P1 | XSS 防护 | AIChat 改用 `SafeMarkdown` 组件（`rehype-sanitize` + `skipHtml` + 禁用危险标签），替代裸 `ReactMarkdown` | `frontend/src/components/SafeMarkdown.tsx`, `AIChat.tsx` |
| P1 | 加密升级 | API Key 加密 salt 从硬编码改为 `SHA256(key)` 派生；PBKDF2 迭代从 100,000 提升至 600,000 | `backend/app/encryption.py` |
| P1 | 传输/Host 安全 | CORS `allow_methods` / `allow_headers` 从 `"*"` 收紧为显式白名单；启用 `TrustedHostMiddleware` | `backend/app/main.py`, `backend/app/config.py` |
| P1 | 代码卫生 | 移除所有 `print(..., file=sys.stderr)` 调试语句，统一使用 `logging` | `backend/app/services/ai_proxy.py` 等 |
| P2 | 密码策略 | 复杂度校验增加特殊字符要求（`!@#$%^&*...`）；`hash_password` 先 SHA-256 预哈希再 bcrypt，规避 72 字节截断 | `backend/app/auth.py` |
| P2 | 请求限制 | 新增 `RequestSizeLimitMiddleware` + `h11_max_request_size=10MB`，防止超大请求 DoS | `backend/app/main.py` |
| P2 | 日志脱敏 | admin 操作日志返回时递归红码敏感字段（`password`, `api_key`, `secret`, `token` 等） | `backend/app/routers/admin.py` |
| P2 | 前端安全 | 移除 `atob()` 客户端 JWT 解析，用户角色/ID 统一由 `/auth/me` 接口返回并缓存 | `frontend/src/services/api.ts`, `frontend/src/App.tsx` |
