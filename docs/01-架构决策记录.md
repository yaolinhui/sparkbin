## 决策列表

### DEC-001：前后端分离架构

**日期**：2026-04-08

**决策内容**：将原本纯前端（localStorage）的 SparkBin 改造为前后端分离架构。

**选项对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| 保持纯前端 + localStorage | 简单，无部署成本 | 数据易丢失，无法多设备同步，AI 配置不安全 |
| 纯前端 + GitHub JSON 备份 | 有云端备份 | 需要暴露 GitHub Token，操作复杂 |
| **前后端分离（选中）** | 数据安全持久化，AI Key 存后端，可扩展用户系统 | 需要维护后端服务，部署成本增加 |

**决策理由**：
- AI 配置（API Key）必须安全存储，不能放在前端
- 需要用户认证和权限管理（admin / user）
- 未来可能需要多设备同步和协作功能

**影响范围**：全部项目架构

### DEC-002：后端技术栈选 FastAPI + SQLAlchemy

**日期**：2026-04-08

**决策内容**：后端使用 Python FastAPI 框架，ORM 使用 SQLAlchemy 2.0，数据库支持 SQLite（开发）和 PostgreSQL（生产）。

**选项对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| Node.js + Express + Prisma | 前后端同语言 | 对 AI 生成代码的友好度不如 Python |
| **FastAPI + SQLAlchemy（选中）** | 现代、类型安全、自动 API 文档、AI 生成质量高 | 需要维护 Python 环境 |
| Django | 功能全、生态成熟 | 过重，不适合轻量级 API 服务 |

**决策理由**：
- FastAPI 的 Pydantic 模型与前端 TypeScript 类型天然对应
- SQLAlchemy 2.0 支持类型提示，AI 不容易写错
- SQLite 适合本地开发，零配置启动

**影响范围**：`backend/` 目录全部代码

### DEC-003：AI 代理模式（后端代理）

**日期**：2026-04-09

**决策内容**：前端不直接调用任何 AI 提供商的 API，所有 AI 请求通过后端代理转发。

**选项对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| 前端直接调用 OpenAI/DeepSeek API | 简单，减少后端负载 | API Key 暴露在前端，安全风险极高 |
| **后端代理模式（选中）** | API Key 安全存储（加密），可统一限流/日志/切换模型 | 增加后端复杂度，需要维护多提供商适配 |

**决策理由**：
- API Key 安全性是底线，绝对不能暴露到前端
- 后端可以统一记录 AI 调用日志（AICallLog）
- 可以在后端实现模型切换、失败重试等逻辑

**影响范围**：`backend/app/services/ai_proxy.py`，`frontend/src/services/ai.ts`

### DEC-004：Brutalist（粗野主义）设计风格

**日期**：2026-04-07

**决策内容**：采用 Brutalist 设计风格——等宽字体（JetBrains Mono）、零圆角边框、高对比度、深色/浅色双主题。

**选项对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| 现代圆润 Material Design | 大众接受度高 | 千篇一律，缺乏个性 |
| 纸张纹理拟物风 | 有质感 | 实现复杂，与深色主题冲突 |
| **Brutalist 粗野主义（选中）** | 独特、高识别度、与"开发者工具"定位契合、实现简单 | 部分用户可能觉得"太硬" |

**决策理由**：
- 目标用户是 Vibe 开发者 / Indie Hacker，喜欢有个性的工具
- Brutalist 风格与终端/代码编辑器的视觉语言一致
- 零圆角 + 等宽字体在 Tailwind 中极易实现

**影响范围**：`frontend/src/index.css`，`frontend/tailwind.config.js`

### DEC-005：SSE 流式响应（而非 WebSocket）

**日期**：2026-04-10

**决策内容**：AI 聊天使用 SSE（Server-Sent Events）实现流式响应，而非 WebSocket。

**选项对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| WebSocket | 双向实时，适合高频交互 | 需要维护连接状态，前后端复杂度增加，防火墙/Nginx 配置麻烦 |
| **SSE（选中）** | 基于 HTTP，简单，自动重连，前端用 EventSource 即可 | 仅服务端向客户端推送，不适合双向高频场景 |

**决策理由**：
- AI 聊天是"一问一答"模式，SSE 完全够用
- SSE 基于标准 HTTP，调试简单（curl 可直接测试）
- 不需要维护 WebSocket 连接状态，减少 bug 来源

**影响范围**：`backend/app/routers/ai.py`（POST /chat），`frontend/src/services/ai.ts`

### DEC-006：六阶段 Vibe 工作流（IDEA → MONETIZE）

**日期**：2026-04-07

**决策内容**：项目生命周期固定为六个阶段：想法(IDEA) → 验证(VALIDATE) → 原型(PROTOTYPE) → 发布(SHIP) → 增长(GROW) → 变现(MONETIZE)。

**选项对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| 自由看板（任意列） | 灵活 | 缺乏引导，新手不知道该怎么走 |
| **固定六阶段（选中）** | 提供清晰的 Indie Hacker 路径，每个阶段有明确目标和退出标准 | 不够灵活，某些项目可能跳过某些阶段 |

**决策理由**：
- 产品定位就是"帮助 Vibe 开发者系统地推进项目"
- 固定阶段提供仪式感，降低决策疲劳
- 每个阶段可以设计专用的编辑器和 AI 辅助逻辑

**影响范围**：`frontend/src/types/index.ts`（StageKey），`backend/app/models.py`，六阶段组件

### DEC-007：命名导出（禁用默认导出）

**日期**：2026-04-07

**决策内容**：项目中全部使用命名导出（`export function Xxx`），禁止使用默认导出（`export default`）。

**决策理由**：
- 命名导出在重构时更容易搜索和替换
- IDE 自动补全更精确
- 避免默认导出导致的重命名混乱

**影响范围**：全部前端 `src/` 代码

### DEC-008：interface 优先于 type

**日期**：2026-04-07

**决策内容**：TypeScript 类型定义优先使用 `interface`，而非 `type`。

**决策理由**：
- interface 支持声明合并，扩展性更好
- interface 错误提示更友好
- 与面向对象风格更一致

**影响范围**：`frontend/src/types/index.ts` 及所有类型定义

### DEC-009：前端状态管理使用 Zustand（而非 Redux）

**日期**：2026-04-08

**决策内容**：前端状态管理使用 Zustand，不使用 Redux 或 Context + useReducer。

**选项对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| Redux + RTK | 生态成熟，调试工具强 | 样板代码多，对 AI 不友好 |
| Context + useReducer | 无需额外依赖 | 性能差，组件树深时 re-render 严重 |
| **Zustand（选中）** | API 极简，无样板代码，TypeScript 友好，AI 容易写对 | 生态不如 Redux 丰富 |

**决策理由**：
- Zustand 的 API 极简，AI 生成的代码不容易出错
- 不需要 Provider 包裹，减少嵌套
- 中间件支持持久化、日志等

**影响范围**：`frontend/src/stores/projectStore.ts`，`frontend/src/stores/aiStore.ts`

### DEC-010：JWT Token 存储于 localStorage

**日期**：2026-04-08

**决策内容**：用户认证使用 JWT，Token 存储在浏览器 localStorage 中。

**选项对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| Cookie + HttpOnly | XSS 安全 | 需要处理 CSRF，跨域麻烦 |
| **localStorage + Bearer Token（选中）** | 实现简单，前后端分离友好 | XSS 攻击可窃取 Token |

**决策理由**：
- 简化实现，降低前后端复杂度
- 当前是单用户/小团队工具，安全要求不是最高优先级
- 如未来需要更高安全性，可迁移到 Cookie + Refresh Token 方案

**影响范围**：`frontend/src/services/api.ts`，`backend/app/auth.py`

### DEC-011：阶段原生格式（AI 回复固定四段结构）

**日期**：2026-04-12

**决策内容**：AI 回复项目相关内容时，必须包含固定四段结构：`【阶段事实】`、`【关键缺口】`、`【下一步动作】`、`【可同步JSON】`。

**决策理由**：
- 统一 AI 输出格式，便于前端解析和展示
- 可自动提取 `sync_payload` 同步到编辑器
- 可自动提取 `next_question` 作为下一轮提问建议
- 验证失败时自动重试，保证输出质量

**影响范围**：`backend/app/services/stage_context.py`，`backend/app/routers/ai.py`

### DEC-012：Stripe 测试模式集成

**日期**：2026-04-13

**决策内容**：支付功能仅集成 Stripe 测试模式，不接入真实支付。

- 避免真实交易带来的法律和税务复杂性

### DEC-013：数据库迁移使用 Alembic，但禁止修改已有迁移文件

**日期**：2026-04-08

**决策内容**：数据库迁移使用 Alembic，但已创建的迁移文件禁止修改。如有 schema 变更，新增迁移文件。

**决策理由**：
- 修改已有迁移会导致团队环境不一致
- SQLite 开发环境对 ALTER TABLE 支持有限，新增迁移更安全
- 启动时动态添加缺失列作为兼容性补丁（`backend/app/main.py` 中的 `init_database`）

**影响范围**：`backend/alembic/versions/`

### DEC-014：前端路由使用 BrowserRouter（而非 HashRouter）

**日期**：2026-04-10

**决策内容**：前端路由使用 `BrowserRouter`，配合 Vite 的 `historyApiFallback`。

**决策理由**：
- URL 更美观（`/project/123` 而非 `/#/project/123`）
- 利于 SEO（虽然本项目不需要）
- Vite dev server 和生产构建都支持 history fallback

**影响范围**：`frontend/src/App.tsx`，`frontend/vite.config.ts`

---

### DEC-015：前端构建产物分块策略（manualChunks）

**日期**：2026-04-12

**决策内容**：Vite 构建使用 `rollupOptions.output.manualChunks` 手动分块，将大型依赖拆分为独立 chunk。

**决策理由**：
- react-force-graph、reactflow、@tiptap、@dnd-kit 等库体积大，独立加载减少首屏时间
- 缓存策略更优：第三方库变化少，可长期缓存

**影响范围**：`frontend/vite.config.ts`
