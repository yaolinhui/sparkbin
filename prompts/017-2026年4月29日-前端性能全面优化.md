# 任务 #28：前端性能全面优化

## 问题描述

经性能排查，SparkBin 前端存在严重的运行时卡顿和首屏加载慢问题，影响用户体验。核心瓶颈集中在 Bundle 体积过大、React 高频重渲染、以及网络请求缺乏优化。

### 1. 首屏加载慢 — 主包 548KB 超标
- **根因**：`App.tsx` 中所有页面组件同步导入，路由无懒加载。`vite.config.ts` 的 `manualChunks` 仅拆分了 node_modules，所有页面/组件代码被打入主 chunk。
- **修复**：路由页面改用 `React.lazy()` 动态导入；在 `manualChunks` 中将大型库（react-markdown）拆出。

### 2. AIChat 流式消息高频重渲染 — 运行时最卡顿的交互
- **位置**：`src/components/AIChat.tsx:393-399`
- **根因**：SSE 每收到一个 chunk 就执行 `setMessages(...)`，导致整个 AIChat 组件（含消息列表、输入框、工具栏）每 50-100ms 重渲染一次。
- **修复**：使用 `useRef` 暂存流式内容，流结束或每 200ms 节流后再更新 state。

### 3. ProjectCard 无 memo — 看板交互卡顿
- **位置**：`src/components/ProjectCard.tsx:37`
- **根因**：普通函数组件，父组件 ProjectBoard 任何状态变更都触发所有卡片重渲染。
- **修复**：`export const ProjectCard = memo(function ProjectCard(...) { ... })`

### 4. ProjectBoard useEffect 依赖不当
- **位置**：`src/components/ProjectBoard.tsx:115-164`
- **根因**：`useEffect` 依赖数组包含 `showToast`（来自 useToast，引用不稳定）和 `petConfigKey`（每次渲染重新计算），导致 mount 后可能重复执行。
- **修复**：移除不稳定的依赖，将 `petConfigKey` 用 `useMemo` 固定，`showToast` 从依赖中移除（可用 ref 保存）。

### 5. API 请求无缓存 — 重复网络开销
- **位置**：`src/services/api.ts:206-282`
- **根因**：`request` 函数无 GET 请求去重/缓存。`authApi.getMe()` 在 App.tsx、ProjectBoard 中被重复调用。
- **修复**：为 GET 请求添加简单的内存缓存（3 秒 TTL + 相同请求去重）。

### 6. ProjectDetail 无请求取消
- **位置**：`src/components/ProjectDetail.tsx:467-519`
- **根因**：快速切换项目时，旧请求未取消，可能与新请求并发。
- **修复**：在 useEffect cleanup 中设置取消标志，或使用 AbortController。

### 7. date-fns 全量导入
- **位置**：`src/components/MonthView.tsx:3`
- **根因**：`import { format, startOfMonth, ... } from 'date-fns'` 可能引入整个库（取决于 tree-shaking）。
- **修复**：改为按需导入 `import format from 'date-fns/format'` 等形式。

### 8. RichTextEditor useEditor 配置不稳定
- **位置**：`src/components/RichTextEditor.tsx:32-43`
- **根因**：`extensions` 数组和 `finalPlaceholder` 在每次渲染时都是新引用，可能导致 Tiptap 内部重新初始化。
- **修复**：将 `finalPlaceholder` 用 `useMemo` 包裹，extensions 数组提取到组件外部（或用 `useMemo`）。

### 9. SafeMarkdown 同步导入增加主包体积
- **位置**：`src/components/AIChat.tsx:3`（导入 SafeMarkdown）
- **根因**：`react-markdown` 及其依赖被打入主包。
- **修复**：SafeMarkdown 组件本身保持同步导出，但通过 `manualChunks` 将 `react-markdown` 拆分为独立 chunk。

### 6. `frontend/src/services/api.ts`（GET 请求缓存）

**允许修改：**
- `request` 函数：为 GET 请求添加内存缓存和请求去重

**具体修改：**
在文件顶部添加：
```typescript
// 简单的 GET 请求内存缓存（3 秒 TTL）
interface CacheEntry {
  data: unknown;
  timestamp: number;
  promise: Promise<unknown> | null;
}
const _apiCache = new Map<string, CacheEntry>();
const _CACHE_TTL_MS = 3000;
```

在 `request` 函数中（第 206-282 行），在构造请求前添加：
```typescript
async function request<T>(
  endpoint: string,
  options: RequestInit & { __retry?: boolean } = {}
): Promise<T> {
  const url = `${API_BASE_URL}${endpoint}`;
  const isRetry = options.__retry ?? false;
  const method = options.method || 'GET';

  // GET 请求缓存（非重试请求）
  if (method === 'GET' && !isRetry) {
    const cached = _apiCache.get(url);
    if (cached && Date.now() - cached.timestamp < _CACHE_TTL_MS) {
      return cached.data as T;
    }
    // 请求去重：如果已有进行中的相同请求，等待其结果
    if (cached?.promise) {
      return cached.promise as Promise<T>;
    }
  }

  // ... 原有 headers 构造逻辑 ...

  // 发起请求
  const doRequest = async (): Promise<T> => {
    const response = await fetch(url, { ...options, headers });
    // ... 原有响应处理逻辑 ...
    return response.json();
  };

  if (method === 'GET' && !isRetry) {
    const promise = doRequest().then((data) => {
      _apiCache.set(url, { data, timestamp: Date.now(), promise: null });
      return data;
    }).catch((err) => {
      _apiCache.delete(url);
      throw err;
    });
    _apiCache.set(url, { data: null, timestamp: Date.now(), promise });
    return promise;
  }

  return doRequest();
}
```

- POST/PUT/DELETE 请求逻辑（不缓存）
- 401 自动刷新逻辑

## 测试方案

1. **路由懒加载测试**：
   - 首次访问首页，Network 面板中不应加载 ProjectDetail/AdminPage 的 chunk
   - 点击项目进入详情页后，应看到 `project-detail-xxx.js` 被动态加载

2. **AIChat 节流测试**：
   - 打开 AI Chat，发送消息
   - React DevTools Profiler 中观察 AIChat 重渲染频率，应降至每 200ms 一次（而非每 chunk 一次）
   - 流式输出内容应仍然实时可见（无明显延迟感）

3. **ProjectCard memo 测试**：
   - React DevTools Profiler：在 ProjectBoard 中切换 filter 或打开 modal，ProjectCard 不应全部重渲染（仅状态变化的卡片或父级重渲染）

   - 连续 2 秒内调用 `authApi.getMe()` 两次，Network 面板中应只看到 1 个请求

   - 快速在项目列表中切换不同项目，Network 面板中不应出现多个并发的 `/projects/{id}` 请求

6. **构建体积测试**：
   - `npm run build` 后主包（index chunk）应小于 400KB（原 548KB）
   - 无 TypeScript 编译错误

## 关键注意事项

1. **路由懒加载与 Suspense fallback**：fallback UI 的风格必须与现有 Brutalist 设计一致（黑白色调、JetBrains Mono、零圆角、加载动画）。
2. **AIChat 节流延迟**：200ms 是经验值，如果用户感觉流式输出有"卡顿感"，可降至 100ms。但不可低于 50ms，否则失去优化意义。
3. **API 缓存仅针对 GET**：POST/PUT/DELETE 绝不能缓存，否则会导致数据不同步。
4. **ProjectBoard useEffect 的 eslint-disable**：这是有意的——我们明确只需要 mount 时执行一次，而非跟随任何依赖变化。请在注释中说明原因。
5. **date-fns 按需导入**：如果 Vite 的 tree-shaking 已经在做按需裁剪，此修改的收益可能不大（几 KB），但仍是良好实践。
6. **Tiptap StarterKit**：本次优化不涉及将 StarterKit 拆分为单独扩展（风险较高，需修改 package.json 依赖，建议作为后续迭代）。
