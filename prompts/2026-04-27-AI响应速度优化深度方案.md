# AI 响应速度优化深度方案

> **调研日期**：2026-04-27
> **问题**：用户报告"每个会话访问 AI 的速度都太慢"
> **约束**：继续使用 DeepSeek 模型，Kimi API 当前不可用
> **目标**：将用户感知延迟从"十几秒空等"降低到"两三秒出首字"

---

## 一、根因分析（5 层）

### 🔴 P0：后端在"吞掉"流式响应（架构缺陷）

**位置**：`backend/app/routers/ai.py` 第249-304行，`event_generator()`

**现状逻辑**：
```
1. await _collect_sse_chunks()      ← 阻塞，等 AI 生成完整回复
2. _extract_content_from_sse_chunks() 
3. validate_stage_native_response()  ← 格式验证
4. if 格式不对: 再次调用 AI (retry) ← 可能二次阻塞
5. yield meta_event                  ← 这时前端才开始收数据
6. for chunk in selected_chunks: yield chunk
```

**致命问题**：SSE（Server-Sent Events）的设计目的是"边生成边推送"，但当前实现把流完全缓存到内存后才推送给前端。用户点击发送后，前端**完全收不到数据**，直到 AI 生成完整回复（10-20秒）。

**用户感知**：点了发送 → 卡住很久 → 突然蹦出一大段文字。

### 🟠 P1：stage_loop retry 导致 AI 被调用两次

**位置**：同文件第255-275行

当 `enable_stage_loop=true`（前端默认）时：
- AI 第一次回复缺少固定章节 → 后端构造修复提示词 → **再次调用 AI**
- 延迟直接翻倍（20秒 → 40秒）

### 🟡 P2：系统提示词过长，token 暴增

**位置**：`backend/app/services/stage_context.py` 第333-357行

`build_stage_native_system_prompt()` 把完整的 `stage.content`（原始 JSON）通过 `json.dumps(snapshot)` 塞进 prompt：

```python
f"阶段快照(JSON): {json.dumps(snapshot, ensure_ascii=False)}"
```

`snapshot` 包含 `content.raw`，即阶段的完整原始数据。如果 Grow 阶段有大量内容日历，prompt 长度可能达到数千 token。

**后果**：
- DeepSeek 处理长 prompt 更慢
- 首字延迟（TTFT）显著增加
- API 费用增加（虽然用户说 Token 不限量，但模型处理长文本本身就更慢）

### 🟡 P3：DeepSeek API key 已失效

**证据**：`ai_call_logs` 最近记录：
```
Authentication Fails, Your api key: ****2345 is invalid
```

**后果**：
- 偶发请求失败，需要用户重新配置 key
- 失败时无 fallback（`chat_completion` 没有 fallback，只有 `generate_idea_suggestions` 有）

### 🟢 P4：缺少 TTFT 监控

当前 `AICallLog` 只记录 token 数量和状态，不记录"从发送到首字"的时间。无法量化优化效果。

## 二、修复方案（4 项）

### 修复1：改为真正的流式转发（最关键）

**目标**：让前端在 AI 返回第一个 chunk 时立即收到数据。

**修改文件**：`backend/app/routers/ai.py`

**新逻辑**：
```python
async def event_generator():
    collected_chunks = []
    text_buffer = []

    # 边收边发
    generator = ai_service.chat_completion(...)
    async for chunk in generator:
        collected_chunks.append(chunk)
        yield chunk
        # 同时提取文本用于后续验证
        for line in chunk.splitlines():
            ...

    full_text = "".join(text_buffer)

    # 流结束后做验证和提取（不再 retry）
    sync_payload = extract_sync_payload(full_text)
    ...
    yield meta_event
    yield "data: [DONE]\n\n"
```

**影响**：
- ✅ 用户感知延迟从"完整生成时间"降到"首字延迟"
- ✅ 不再 retry，避免二次调用
- ⚠️ meta event 从"内容之前"变为"内容之后"
  - 前端 `onMeta` 会在流结束后触发，不影响功能

### 修复2：优化系统提示词长度

**目标**：减少 prompt token 数量，降低模型处理时间。

**修改文件**：`backend/app/services/stage_context.py`

**改动**：
- 用 `minimal_snapshot` 替代完整 `snapshot`，只保留：项目标题、当前阶段、完成度、缺口列表
- 删除 `content.raw`（原始阶段数据）的注入
- 在格式要求中加入"缺一不可"，强化格式约束

**预期效果**：prompt 长度减少 30%-70%（取决于阶段内容大小）。

### 修复3：移除 retry 逻辑

**理由**：
- retry 会触发二次 AI 调用，延迟翻倍
- 更好的方案是用更强的 prompt engineering 让 AI 一次答对
- 即使偶尔格式不对，前端可以优雅处理（不显示同步按钮即可）

### 修复4：添加性能监控

**目标**：量化 TTFT 优化效果。

**修改文件**：`backend/app/services/ai_proxy.py`

**改动**：在 `chat_completion` 中记录：
- `first_chunk_at`：从发送请求到收到第一个 chunk 的时间
- 将 TTFT 写入 `AICallLog` 的扩展字段（或日志）

## 三、测试方案

### 测试1：TTFT 对比测试

**脚本**：`backend/tests/test_ai_chat_perf.py`

**方法**：
1. Mock `chat_completion` 的 generator，模拟延迟 5 秒的 AI 响应
2. 调用 `/ai/chat` 端点
3. 测量：从请求发送到收到第一个 SSE chunk 的时间
4. 测量：从请求发送到收到 [DONE] 的时间

**预期结果**：
- 修复前：TTFT ≈ TBT（Total Blocking Time），约 5 秒
- 修复后：TTFT < 0.1 秒（本地处理），TBT ≈ 5 秒

### 测试2：Prompt 长度测试

**方法**：
1. 构造一个大内容的项目（模拟 Grow 阶段 100 条日历）
2. 调用 `build_stage_native_system_prompt()`
3. 统计返回字符串长度

**预期结果**：
- 修复前：prompt 长度 ~3000-5000 字符
- 修复后：prompt 长度 ~500-800 字符

### 测试3：流式响应完整性测试

**方法**：
1. 发送聊天请求
2. 收集所有 SSE chunk
3. 验证：内容 chunk 在 meta event 之前到达
4. 验证：meta event 包含正确的 `sync_payload` 和 `next_question`
5. 验证：最后以 `[DONE]` 结束

## 五、风险提示

1. **meta event 时序变化**：当前前端期望 meta 在内容之前。修复后 meta 在内容之后。经代码审查，前端 `onMeta` handler 不依赖时序，只负责更新 UI 状态，所以安全。
2. **移除 retry 的副作用**：AI 偶尔格式不对时，`sync_payload` 可能为空。前端已有空值处理逻辑（不显示同步按钮）。
3. **精简 prompt 的副作用**：AI 看不到完整阶段内容，可能回答不够精准。但 `build_stage_snapshot` 的 `evaluation.evidence` 已经包含关键摘要，足够 AI 理解上下文。
