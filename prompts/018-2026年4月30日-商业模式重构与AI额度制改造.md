# Prompt：商业模式重构 —— 从订阅制到「免费无限 + AI 额度预付费」

> **任务序号**：018
> **创建时间**：2026年4月30日
> **目标**：将 SparkBin 的商业模式从"按月订阅 + 功能限制"重构为"项目无限免费 + AI 调用按量预付费"

---

## 一、背景与决策

### 1.1 当前问题
- 订阅制（$9/月）造成高信任门槛：用户还没体验到价值就要付费决策
- 项目数量限制（免费版3个）阻碍了核心产品价值的传播
- AI 调用按月重置（500次/月）导致月末焦虑和资源浪费

### 1.2 新商业模式
| 层级 | 项目数量 | AI 调用 | 费用 |
|------|---------|---------|------|
| **免费层** | 无限 | 注册赠送 20 次 | 永远免费 |
| **付费层** | 无限 | 预付费额度包 | $5/$10/$20 充值 |

**核心原则**：
1. **项目永远是免费的** —— 项目管理是核心产品价值，不应收费
2. **AI 是付费点** —— 每次 AI 对话消耗真实 API 成本，按量收费合理
3. **预付费制** —— 用户买"AI 对话次数"，不是订阅，余额长期有效
4. **自托管完全免费** —— 自托管用户走本地 Ollama，AI 也免费

### 1.3 版本架构（单一代码库）
```
GitHub 仓库（main 分支）
├── 同一套代码，三个运行模式通过环境变量区分
│
├── 自托管版（用户自己部署）
│   └── ENABLE_PAYMENTS=false → 无充值入口，AI 走 Ollama
│
└── SaaS 版（我们运营 sparkbin.dev）
    └── ENABLE_PAYMENTS=true → 有充值入口，AI 走我们的代理，扣除用户 credits
```
**禁止代码分叉（fork）**。禁止创建 self-hosted / saas 分支。所有差异通过 Feature Flags 和环境变量控制。

## 二、数据库迁移（Alembic）

### 2.1 新增表：credit_transactions（额度流水）
用于记录充值和消费，支持审计和对账。

```python
class CreditTransaction(Base):
    __tablename__ = "credit_transactions"

    id = Column(UUID(as_uuid=True), primary_key=True, default=uuid.uuid4)
    user_id = Column(UUID(as_uuid=True), ForeignKey("users.id"), nullable=False, index=True)
    type = Column(String(20), nullable=False)  # "grant" | "purchase" | "consume" | "refund"
    amount = Column(Integer, nullable=False)   # 正数=增加，负数=扣除
    balance_after = Column(Integer, nullable=False)  # 变动后的余额
    description = Column(String(255), nullable=True)  # "注册赠送" | "充值 $5" | "AI 调用扣除"
    reference_id = Column(String(255), nullable=True)  # Stripe session_id 或 AI call log id
    created_at = Column(DateTime, default=datetime.utcnow, nullable=False)

    user = relationship("User")
```

### 2.2 users 表字段变更
- **新增**：`ai_credits` (Integer, default=20, nullable=False) —— 当前可用 AI 额度
- **新增**：`ai_credits_total_consumed` (Integer, default=0, nullable=False) —— 累计消耗（统计用）
- **保留但废弃**：`subscription_status`, `stripe_customer_id`, `stripe_subscription_id`, `current_tier_id` —— 保留字段兼容旧数据，但逻辑上不再使用
- **移除逻辑限制**：`projects_limit` 逻辑上对所有用户视为 `null`（无限），但数据库字段保留

### 2.3 数据迁移策略
- 所有现有用户的 `ai_credits` 初始化为 20（注册赠送额度）
- 产品尚未上线，无老用户兼容负担

### 3.4 AI 调用路由改造（routers/ai.py）

每次 AI 调用前（stream 和非 stream 都要）：
1. 如果 `ENABLE_PAYMENTS=false`，跳过额度检查（自托管模式）
2. 如果 `ENABLE_PAYMENTS=true`：
   - 检查用户 `ai_credits > 0`
   - 如果余额为 0，返回 402 Payment Required：`{"detail": "AI credits exhausted", "credits": 0}`
   - 调用成功后：
     - `user.ai_credits -= 1`
     - `user.ai_credits_total_consumed += 1`
     - 写入 `credit_transactions` type="consume", amount=-1, description="AI 对话消耗"
     - 将 `credit_transaction_id` 关联到 `ai_call_logs` 记录（新增外键字段或记录 reference_id）

**注意**：一次 AI 对话（stream 或普通）只扣 1 个 credit，不按 token 计费。简化用户心智模型。

## 六、关键架构决策

### 6.1 为什么不按 Token 计费？
- Token 是技术术语，用户不懂 "1K tokens" 是什么意思
- 每次对话扣 1 次，心智模型极其简单
- 我们的成本足够低（Kimi/DeepSeek ~$0.001/次），即使高频用户也有足够毛利

### 6.2 为什么余额长期有效？
- 预付费余额不过期，降低用户购买决策门槛
- "买了不会浪费" 比 "按月清零" 更能促进付费
- 参考 OpenAI / Vercel / Cloudflare 的预付费模式

### 6.3 自托管如何完全免费？
- 自托管用户 `ENABLE_PAYMENTS=false`
- 前端通过 `/auth/me` 的 `enable_payments` 字段判断，不显示充值 UI
- AI 走本地 Ollama，无外部 API 成本
- 这才是真正的"开源免费"——代码全开源，自托管零限制

### 6.4 老订阅用户如何处理？
- 不自动退款，将当前活跃订阅的剩余价值折算为 credits
- 例如：用户本月已付 $9，还剩 15 天 → 折算为 250 credits
- 此后不再有订阅扣费

## 十、循环的设计（AI 阶段推进循环）

### 10.1 背景
当前 AIChat 已有 `enableStageLoop` 和 `nextQuestion` 的雏形，但交互不完整。用户完成一轮对话后，AI 提取"下一轮问题"，但用户需要手动点击"采用并发送"才能继续——这个断点阻碍了循环的流畅性。

### 10.2 目标
设计一个**自动推进的 AI 教练循环**，让 AI 在一个阶段内主动引导用户完成目标：

```
用户输入 → AI 回答 + 阶段快照分析 → 自动生成下一轮问题
    ↑                                                      ↓
    └────────── 用户一键继续 / 或输入新内容 ───────────────┘
```

### 10.3 具体设计

1. **阶段完成度可视化**：
   - AI 回答后，`stageSnapshot.completion.score` 实时更新进度条
   - `missing_items` 列表展示当前阶段还缺什么
   - 当 `score >= 80` 且用户确认后，AI 建议"进入下一阶段"

2. **下一轮问题的智能呈现**：
   - AI 流式输出结束后，如果返回了 `next_question`，在输入框上方显示一个"继续追问"气泡
   - 用户点击后**自动发送**该问题，无需手动粘贴到输入框
   - 如果用户自己输入了新内容，则取消追问气泡

3. **循环结束条件**：
   - 阶段完成度达到 100% → AI 提示"本阶段已完成，是否进入下一阶段？"
   - 用户明确说"结束对话"或"我不需要了" → 循环终止
   - 余额不足（credits = 0）→ 循环终止，引导充值

4. **同步到阶段面板**：
   - AI 回答中的 `sync_payload_structured` 自动解析为可同步内容
   - 在每条 AI 消息下方显示"同步到左侧面板"按钮（当前已有）
   - 循环推进时，如果检测到有实质内容产出，主动询问"是否将内容同步到当前阶段？"

### 10.4 前端交互改造

- `AIChat.tsx`：流式输出结束后，若 `nextQuestion` 存在，在输入框上方渲染 `NextQuestionBubble` 组件
- `NextQuestionBubble`：显示问题文本 + "继续追问"按钮，3 秒无操作自动淡出
- 新增 `handleAutoContinue(question: string)`：自动发送下一轮问题，效果等同于用户手动输入

## 十一、循环的优化（Credits 状态同步循环）

### 11.1 问题
当前 AI 调用扣费后，前端余额显示不会实时更新。用户发送第 20 条消息后余额变为 0，但界面上可能还显示"剩余 1 次"，导致下一条消息发送失败（402）。

### 11.2 目标
优化"AI 调用 → 扣费 → 余额更新 → UI 刷新"的状态同步循环，确保余额显示 100% 准确。

### 11.3 优化方案

1. **乐观更新 + 兜底校验**：
   - 前端发送 AI 请求前，`credits` 乐观减 1（立即更新 UI）
   - AI 流式输出结束后，调用 `paymentsApi.getCreditsStatus()` 获取真实余额，校正差异
   - 如果后端返回 402（余额不足），立即弹出充值弹窗

2. **轮询兜底**：
   - 每 30 秒轮询一次 `credits-status`（仅在 AIChat 打开时）
   - 用户从其他页面切回时，立即刷新余额

3. **充值后自动刷新**：
   - Stripe Checkout 成功回调（`?payment_success=1`）时，立即刷新余额并显示 toast

4. **余额临界预警**：
   - `credits <= 5`：AIChat 头部余额标签变为黄色
   - `credits <= 0`：输入框禁用，显示"余额不足，点击充值"替代 placeholder

### 11.4 代码要点

```typescript
// AIChat.tsx
const [credits, setCredits] = useState<number | null>(null);

// 发送消息前乐观更新
const sendMessage = async (...) => {
  if (credits !== null && credits > 0) {
    setCredits(credits - 1); // 乐观更新
  }
  // ... 调用 AI API
};

// 流结束后真实校验
useEffect(() => {
  if (!isThinking) {
    paymentsApi.getCreditsStatus().then(status => setCredits(status.credits));
```
