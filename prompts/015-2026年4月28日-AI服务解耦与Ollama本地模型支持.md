# 任务 #15：AI 服务解耦与 Ollama 本地模型支持

## 目标
将 SparkBin 的 AI 调用从"硬编码 OpenAI API"改造为"可插拔 Provider 架构"，支持用户选择本地 Ollama 模型，满足数据主权和离线部署需求。

## 现状分析
- 前端 `frontend/src/services/ai.ts`：有 `AIProvider = 'deepseek' | 'kimi' | 'doubao' | 'openai'` 类型，但 UI 层面选择后全部走后端代理
- 后端 `backend/app/services/ai_proxy.py`：`AIProxyService` 通过 `get_active_config()` 读取数据库中的 `AIConfig`，目前仅支持 DeepSeek/Kimi/豆包/OpenAI 四家云端 API
- 后端 `backend/app/models.py`：`AIProvider` enum 只有 4 个值
- 关键问题：自托管用户如果不上传云端 API Key，AI 功能完全无法使用；且项目数据必须外传

## 执行步骤

### 1. 后端：扩展 AI Provider 枚举
- 文件：`backend/app/models.py`
- 在 `AIProvider` enum 中新增 `OLLAMA = "ollama"`
- `AIConfig` 模型兼容：Ollama 不需要 API Key（或填任意占位符），`base_url` 默认 `http://localhost:11434`

### 2. 后端：AI 代理服务兼容 Ollama API
- 文件：`backend/app/services/ai_proxy.py`
- `AIProxyService.chat_completion()` 中增加 Ollama 分支：
  - Ollama 的 API 路径是 `/api/chat`（非 `/v1/chat/completions`），请求/响应格式不同
  - 或者通过 Ollama 的 OpenAI 兼容模式（`/v1/chat/completions`）直接复用现有逻辑
  - **推荐**：使用兼容模式，只需在 `base_url` 后拼接 `/v1`，复用现有 httpx 流式逻辑
- `generate_idea_suggestions()` 等结构化输出方法：Ollama 本地模型（如 llama3.2、qwen2.5）也能生成 JSON，无需额外改造
- `init_default_ai_configs()` 中新增 Ollama 默认配置（默认不启用）

### 3. 后端：新增 Ollama 模型管理端点
- 文件：`backend/app/routers/ai.py`（新建端点或扩展现有）
- 新增 `GET /ai/ollama/models` — 管理员可用，调用 Ollama 的 `/api/tags` 列出本地可用模型
- 新增 `POST /ai/ollama/pull` — 触发模型拉取（可选，高级功能）
- 权限：仅 `admin` 角色可访问

### 4. 后端：AI 配置初始化逻辑调整
- 文件：`backend/app/main.py` 或 `backend/app/services/ai_proxy.py`
- `init_default_ai_configs()` 为 Ollama 添加默认项：
  - `base_url`: `http://ollama:11434`（Docker 网络内）或 `http://localhost:11434`（裸机）
  - `model`: `llama3.2`（轻量、速度快、适合创意辅助）
  - `is_active`: False

### 5. 前端：AI Provider 选择 UI 增加 Ollama
- 文件：`frontend/src/services/ai.ts`
- 在 `AIProvider` 类型中新增 `'ollama'`
- 在 `AI_PROVIDER_NAMES` 中新增 `ollama: 'Ollama (本地)'`
- 在设置/管理后台中，当选择 Ollama 时：
  - 隐藏 "API Key" 输入框（或显示"无需 API Key"）
  - 显示 "Base URL" 输入框（默认 `http://localhost:11434`）
  - 显示模型下拉选择（调用后端 `/ai/ollama/models` 获取本地模型列表）

### 6. 前端：AI 设置页面适配
- 文件：`frontend/src/components/` 中管理 AI 设置的组件（查找现有管理后台）
- 如果当前 AI 管理 UI 是硬编码 4 个 provider 的卡片，改为动态渲染
- Ollama 配置卡片上增加"检测连接"按钮，调用后端测试连通性

### 7. Docker Compose 可选服务
- 文件：`docker-compose.yml`（与任务 #14 联动）
- 新增注释掉的 `ollama` 服务：
  ```yaml
  ollama:
    image: ollama/ollama
    volumes:
      - ollama:/root/.ollama
    # 如需 GPU 加速可添加 runtime: nvidia
  ```
- 在 `SELF_HOSTING.md` 中增加"启用本地 AI"章节：
  - 取消注释 ollama 服务
  - 启动后进入容器执行 `ollama pull llama3.2`
  - 在管理后台启用 Ollama provider
  - 说明：纯离线运行，项目数据不出服务器

## 约束条件

**允许修改：**
- `backend/app/models.py`（新增 enum 值）
- `backend/app/services/ai_proxy.py`（Ollama 兼容逻辑）
- `backend/app/routers/ai.py`（新增端点）
- `backend/app/main.py`（初始化默认配置）
- `frontend/src/services/ai.ts`（新增 provider 类型和名称）
- AI 设置相关的前端组件（动态渲染 provider 卡片）
- `docker-compose.yml`（新增注释掉的 ollama 服务）

**禁止修改：**
- 现有的 DeepSeek/Kimi/豆包/OpenAI 调用逻辑（保持 100% 兼容）
- `backend/alembic/versions/` 下已有迁移文件
- 前端 AI 聊天核心逻辑（`chatCompletionStream` 等）

## 测试方案
1. 启动本地 Ollama（或 Docker ollama 服务），确保 `http://localhost:11434` 可访问
2. 管理后台启用 Ollama provider，配置 base_url
3. 点击"测试连接" → 返回成功
4. 前端切换 AI provider 为 Ollama
5. 进入项目 Idea 阶段，点击"AI 建议" → 正常返回建议内容
6. 断网环境下（仅本地 Ollama），AI 功能仍然可用
7. 切换回 DeepSeek/Kimi → 云端功能正常

## 提交标准
- Ollama 与云端 provider 可以无缝切换
- 不引入新的重量级 SDK（Ollama 通过标准 HTTP/OpenAI 兼容 API 接入）
- `npm run build` 无 TypeScript 错误
- 后端 `python -c "import app.main; print('ok')"` 无语法错误
