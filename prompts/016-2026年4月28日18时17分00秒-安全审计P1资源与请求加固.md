# 任务 #27：安全审计 P1 资源控制与请求安全加固

## 问题描述

经安全审计，发现 6 个 P1 级中危漏洞，集中在资源耗尽防护、外部请求安全和数据脱敏。这些问题可导致 DoS、SSRF、信息泄露和凭证残留风险。

### P1-9：AI 想法建议缓存无大小限制 — 内存耗尽 DoS
- **位置**：`backend/app/services/ai_proxy.py:19-20`
- **根因**：`_idea_suggestion_cache` 是全局字典，无最大容量限制、无 LRU 淘汰。攻击者通过发送大量不同参数请求填满内存。
- **修复**：实现有容量上限的 LRU 缓存（限制 128 条），并限制缓存键的最大长度。

### P1-10：查询参数 `limit` 缺少上限 — DoS / 数据泄露
- **位置**：`backend/app/routers/ai.py:440-464` (`/ai/call-logs`)、`backend/app/routers/admin.py:44-66` (`/admin/logs`)
- **根因**：`limit` 参数无最大值限制，攻击者可传入 `limit=9999999` 导致数据库负载过高和超大响应体。
- **修复**：两个端点均限制 `limit` 最大值为 1000。

### P1-11：外部 HTTP 请求未禁用重定向 — SSRF 风险
- **位置**：
  - `backend/app/services/ai_proxy.py:192` (`test_connection`)
  - `backend/app/services/ai_proxy.py:265` (`chat_completion`)
  - `backend/app/services/ai_proxy.py:404` (`generate_promote_suggestions`)
  - `backend/app/services/ai_proxy.py:525` (`_generate_idea_suggestions_single`)
  - `backend/app/services/ai_proxy.py:668` (`generate_validate_suggestions`)
- **根因**：`httpx.AsyncClient` 默认 `follow_redirects=True`。如果管理员配置的 AI `base_url` 被篡改，攻击者可通过 302 重定向访问内网服务。
- **修复**：所有外部 AI API 调用显式设置 `follow_redirects=False`。

### P1-12：HTML 邮件模板未转义用户输入 — 邮件客户端 XSS
- **位置**：`backend/app/email.py:57-68` (send_verification_email)、`email.py:87-98` (send_password_reset_email)
- **根因**：邮件 HTML 模板直接通过 f-string 拼接 `username`，未进行 HTML 实体转义。如果用户注册时设置用户名为 `<img src=x onerror=alert(1)>`，验证邮件会包含恶意代码。
- **修复**：对 `username` 使用 `html.escape()` 转义。`verify_url` / `reset_url` 由后端生成，相对可控，但为安全起见也一并转义。

### P1-13：日志敏感字段脱敏不完整
- **位置**：`backend/app/routers/admin.py:13-17`
- **根因**：`_SENSITIVE_KEYS` 只包含精确匹配的 key（如 `"password"`），但日志中可能出现 `old_password`、`new_password`、`current_password` 等变体，均不会被脱敏。
- **修复**：引入正则模式匹配，对包含 `password`、`secret`、`token`、`api_key` 等关键词的字段进行模糊匹配脱敏。

### P1-14：OAuth Token 通过 URL Query Param 传递，残留在浏览器历史和 Referrer 中
- **位置**：`frontend/src/App.tsx:32-37`、`backend/app/routers/auth.py:560-566`
- **根因**：OAuth 成功后，后端将 `access_token` 和 `refresh_token` 作为 URL query parameter 重定向回前端。这些 token 会留在浏览器历史、服务器访问日志和 HTTP Referer 中。
- **修复**：后端改为使用 URL fragment（`#access_token=...`）传递；前端 `OAuthHandler` 改为读取 `window.location.hash`。

**1.1 替换无界缓存为有界 LRU 缓存：**
```python
# backend/app/services/ai_proxy.py
from typing import Any

class _LRUCache:
    def __init__(self, capacity: int = 128):
        self.capacity = capacity
        self._cache: dict = {}
        self._order: list = []

    def get(self, key: str) -> Any | None:
        if key in self._cache:
            self._order.remove(key)
            self._order.append(key)
            return self._cache[key]
        return None

    def put(self, key: str, value: Any) -> None:
        if key in self._cache:
            self._order.remove(key)
        elif len(self._order) >= self.capacity:
            oldest = self._order.pop(0)
            del self._cache[oldest]
        self._cache[key] = value
        self._order.append(key)

# 替换原来的全局字典
_idea_suggestion_cache = _LRUCache(capacity=128)
_CACHE_TTL = 3600
```

```python
import re

# 替换原有的精确匹配集合
_SENSITIVE_KEYS = {
    "password", "password_hash", "api_key", "api_key_encrypted",
    "secret", "secret_key", "token", "refresh_token", "access_token",
    "authorization", "cookie", "session",
}
_SENSITIVE_PATTERNS = [
    re.compile(r".*password.*", re.I),
    re.compile(r".*secret.*", re.I),
    re.compile(r".*token.*", re.I),
    re.compile(r".*api[_-]?key.*", re.I),
]

def _is_sensitive_key(key: str) -> bool:
    k = key.lower()
    if k in _SENSITIVE_KEYS:
        return True
    for pattern in _SENSITIVE_PATTERNS:
        if pattern.match(k):
            return True
    return False

def _redact_sensitive(obj):
    if isinstance(obj, dict):
        return {
            k: "***" if _is_sensitive_key(k) else _redact_sensitive(v)
            for k, v in obj.items()
        }
    elif isinstance(obj, list):
        return [_redact_sensitive(v) for v in obj]
    return obj
```

def _oauth_success_redirect(access_token: str, refresh_token: str) -> RedirectResponse:
    settings = get_settings()
    # 使用 fragment 而非 query，避免 token 进入服务器日志和 referrer
    fragment = urlencode({

```typescript
// frontend/src/App.tsx
function OAuthHandler({ onLogin }: { onLogin: (resp?: { access_token: string; refresh_token: string }) => void }) {
  const [, setSearchParams] = useSearchParams();

  useEffect(() => {
    // 从 URL fragment 读取 OAuth 参数
    const hash = window.location.hash.slice(1); // 去掉开头的 #
    const params = new URLSearchParams(hash);
    const oauthSuccess = params.get('oauth_success');
    const accessToken = params.get('access_token');
    const refreshToken = params.get('refresh_token');
    const githubConnect = params.get('github_connect');

    if (oauthSuccess === '1' && accessToken && refreshToken) {
      setAuthToken(accessToken);
      setRefreshToken(refreshToken);
      // 清除 URL fragment
      window.history.replaceState(null, '', window.location.pathname + window.location.search);
      onLogin({ access_token: accessToken, refresh_token: refreshToken });
    }

    if (githubConnect === 'success') {
      sessionStorage.setItem('sparkbin_github_connected', '1');
      window.history.replaceState(null, '', window.location.pathname + window.location.search);
    }
  }, [onLogin]);

  return null;
}
```

## 关键注意事项

1. **LRUCache 键长度**：`cache_input = f"{title}:{pain_point}:{original_idea}:{json.dumps(current_notes, sort_keys=True)}"` 可能很长。虽然 MD5 哈希后键长固定为 32 字符，但原始字符串构造时仍会占用临时内存。这不是安全问题，但值得注意。
2. **OAuth Fragment 与现有登录流程的兼容性**：修改后，OAuth 成功后的回调 URL 从 `/?oauth_success=1&access_token=...` 变为 `/#oauth_success=1&access_token=...`。`BrowserRouter` 不受影响，因为 fragment 不会触发路由变化。
3. **邮件转义不影响纯文本版本**：纯文本邮件（`text_body`）不需要 HTML 转义，保持原样即可。只有 `html_body` 需要转义。
4. **`_SENSITIVE_PATTERNS` 的误报风险**：`.*password.*` 可能会误报合法字段（如 `password_strength`），但这在安全场景下是可接受的（宁可过度脱敏，不可遗漏）。
