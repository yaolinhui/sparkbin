## 一、严重（CRITICAL）—— 必须立即修复

### 1.1 AI 额度"先扣费后服务"，调用失败不退还

**风险**：用户付费后如果 AI 服务返回错误、超时或客户端断开，已扣额度无法退还，造成直接经济损失和用户信任崩塌。

**涉及文件**：
- `backend/app/routers/ai.py:239`（chat_completion）
- `backend/app/routers/ai.py:374`（promote-suggest）
- `backend/app/routers/ai.py:422`（idea-suggest）
- `backend/app/routers/ai.py:447`（validate-suggest）
- `backend/app/routers/surveys.py:184`（survey generate）
- `backend/app/routers/surveys.py:389`（survey analyze）

**修复方案**：
1. 将扣费逻辑改为"预扣费 + 事后结算"模式。
2. 创建一个新的 "pending" 状态的 CreditTransaction，在 AI 调用成功后再确认扣费；如果失败则回滚/退还。
3. 对于流式接口（`/ai/chat`），在成功收到第一个 SSE chunk 后再执行最终扣费确认。
4. 所有 AI 调用接口统一使用此模式。

**具体实现**：
```python
# 在 backend/app/routers/ai.py 中新增一个辅助函数

def _hold_ai_credit(user: User, db: Session, reference_id: str | None = None) -> CreditTransaction:
    """预扣 AI 额度，返回 pending 状态的 transaction"""
    if user.role.value == "admin":
        return None
    if user.ai_credits <= 0:
        raise HTTPException(status_code=402, detail="AI 调用额度已耗尽。")
    
    user.ai_credits -= 1
    user.ai_credits_total_consumed += 1
    tx = CreditTransaction(
        user_id=user.id,
        type="consume",
        amount=-1,
        balance_after=user.ai_credits,
        description="AI 对话消耗（待确认）",
        reference_id=reference_id,
        status="pending",  # 需要给 CreditTransaction 模型新增 status 字段
    )
    db.add(tx)
    db.commit()
    return tx

def _confirm_ai_credit(tx: CreditTransaction | None, db: Session) -> None:
    """确认扣费"""
    if tx is None:
        return
    tx.status = "confirmed"
    tx.description = tx.description.replace("（待确认）", "")
    db.commit()

def _refund_ai_credit(user: User, tx: CreditTransaction | None, db: Session) -> None:
    """调用失败时退还额度"""
    if tx is None:
        return
    user.ai_credits += 1
    user.ai_credits_total_consumed -= 1
    tx.status = "refunded"
    tx.description = tx.description.replace("（待确认）", "（已退还）")
    db.commit()
```

### 1.3 注册即送额度 + 邮箱验证非强制 = 无限注册薅羊毛

**风险**：用户可以批量注册账号，每个账号领取 20 免费额度。IP 限流是内存级的，多 worker 下完全失效。

**涉及文件**：
- `backend/app/config.py:31`
- `backend/app/routers/auth.py:411-455`
- `backend/app/routers/auth.py:756-776`（Google OAuth）
- `backend/app/routers/auth.py:899-942`（GitHub OAuth）

**修复方案**：
1. **邮箱验证强制化**：注册后账号处于 `email_pending` 状态，未验证邮箱前无法使用任何 AI 功能。
2. **注册赠送额度改为"验证后发放"**：只有 `email_verified=True` 的账号才获得免费额度。
3. **增加每日注册总量限制**：基于内存的全局计数器（如 100 个/天），超过则要求邀请码或暂停注册。
4. **OAuth 注册同样要求邮箱验证**：OAuth 用户的邮箱如未验证，同样处于受限状态。
5. **同 IP 注册限制**：24 小时内同一 IP 最多注册 3 个账号。

```python
    email = payload.get("email")

                user_id=user.id,
                type="grant",
                amount=settings.credits_grant_on_register,
                balance_after=user.ai_credits,
                description="邮箱验证赠送",
            )
            db.add(tx)
    
    db.commit()
    return VerifyEmailResponse(success=True, message="邮箱验证成功")
```

### 1.4 内存级限流器无法跨进程共享

**风险**：`uvicorn --workers 4` 或 Kubernetes 多 Pod 部署时，每个进程有独立的限流内存，攻击者可以轮流绕过。

**涉及文件**：
- `backend/app/auth.py:24`（`_auth_attempts`）
- `backend/app/auth.py:29`（`_captcha_store`）
- `backend/app/routers/surveys.py:32`（`_rate_limit_store`）

**修复方案**：
1. 引入 Redis 作为分布式限流存储。
2. 如果 Redis 不可用，降级为内存限流 + 增加进程间同步（共享文件/SQLite 内存表，但不推荐）。
3. 保持向后兼容：Redis 连接失败时自动降级到内存限流，并发出 warning 日志。

**具体实现**：
```python
# backend/app/rate_limiter.py（新增文件）

import os
import redis
import json
from datetime import datetime, timedelta
from typing import Optional

_redis_client: redis.Redis | None = None

def get_redis() -> redis.Redis | None:
    global _redis_client
    if _redis_client is not None:
        return _redis_client
    redis_url = os.environ.get("REDIS_URL")
    if not redis_url:
        return None
    try:
        _redis_client = redis.from_url(redis_url, decode_responses=True)
        _redis_client.ping()
        return _redis_client
    except Exception:
        return None

class RateLimiter:
    def __init__(self, key_prefix: str, max_requests: int, window_seconds: int):
        self.key_prefix = key_prefix
        self.max_requests = max_requests
        self.window_seconds = window_seconds
    
    def is_allowed(self, key: str) -> tuple[bool, int]:
        """返回 (是否允许, 重试等待秒数)"""
        r = get_redis()
        full_key = f"{self.key_prefix}:{key}"
        now = int(datetime.utcnow().timestamp())
        
        if r:
            # Redis 分布式限流
            pipe = r.pipeline()
            pipe.zremrangebyscore(full_key, 0, now - self.window_seconds)
            pipe.zcard(full_key)
            pipe.zadd(full_key, {str(now): now})
            pipe.expire(full_key, self.window_seconds)
            _, count, _, _ = pipe.execute()
            if count >= self.max_requests:
                return False, self.window_seconds
            return True, 0
        else:
            # 降级到内存限流
            return _memory_check(full_key, now)

# auth.py 中的限流函数替换为使用 RateLimiter
```

**风险**：管理员测试 AI 配置时，`base_url` 可指向内网地址，探测内部服务。

```python
# backend/app/services/ai_proxy.py

import ipaddress
from urllib.parse import urlparse

def _is_safe_url(url: str) -> bool:
    parsed = urlparse(url)
    hostname = parsed.hostname
    if not hostname:
        return False
    
    # 禁止 localhost
    if hostname.lower() in ("localhost", "127.0.0.1", "0.0.0.0", "::1"):
        return False
    
    # 检查是否是 IP 地址
    try:
        ip = ipaddress.ip_address(hostname)
        if ip.is_private or ip.is_loopback or ip.is_link_local or ip.is_reserved:
            return False
    except ValueError:
        pass  # 是域名，继续检查
    
    return True
```

### 3.3 问卷公开提交接口的 IP 限流可伪造

**风险**：`_get_client_ip()` 信任 `X-Forwarded-For`，在代理配置不当的情况下可被伪造。

**涉及文件**：
- `backend/app/routers/surveys.py:69-74`

**修复方案**：
1. 增加可信代理白名单配置。
2. 不在可信代理后面的环境中，忽略 `X-Forwarded-For`。
3. 或者：始终使用 `request.client.host`（直接连接 IP）。

**具体实现**：
```python
# backend/app/routers/surveys.py

_TRUSTED_PROXIES = os.environ.get("TRUSTED_PROXIES", "").split(",")

def _get_client_ip(request: Request) -> str:
    """获取客户端真实 IP，仅在可信代理环境下信任 X-Forwarded-For"""
    direct_ip = request.client.host if request.client else "unknown"
    
    # 如果直接连接 IP 不在可信代理列表中，忽略 X-Forwarded-For
    if direct_ip not in _TRUSTED_PROXIES:
        return direct_ip
    
    forwarded = request.headers.get("x-forwarded-for")
    if forwarded:
        return forwarded.split(",")[0].strip()
    return direct_ip
```

### 3.4 AI 调用日志的错误信息可能包含用户敏感内容

**风险**：`AICallLog.error_msg` 可能包含 AI 服务商返回的用户 prompt 片段。

**涉及文件**：
- `backend/app/services/ai_proxy.py:310-311`
- `backend/app/services/ai_proxy.py:438`

**修复方案**：
1. 对 `error_msg` 进行敏感信息扫描，如发现 `api_key`、`Bearer `、`sk-` 等模式，替换为 `[REDACTED]`。
2. 限制 error_msg 只记录 API 服务商的错误码和类型，不记录完整响应体。

### 3.5 OperationLogger 未脱敏即入库

**风险**：`old_values`/`new_values` 可能包含密码、API key 等敏感信息，以明文 JSON 存入数据库。

**涉及文件**：
- `backend/app/services/logger.py:28-29`

**修复方案**：
1. 将 `admin.py` 中的 `_redact_sensitive()` 逻辑提取到公共模块。
2. 在 `OperationLogger.log()` 中写入数据库前执行脱敏。

**具体实现**：
```python
# backend/app/services/logger.py
from .sensitive_redactor import redact_sensitive  # 从 admin.py 提取

def log(self, ...):
    log = OperationLog(
        user_id=user_id,
        action=action,
        entity_type=entity_type,
        entity_id=entity_id,
        old_values=json.dumps(redact_sensitive(old_values), ensure_ascii=False, default=str) if old_values else "",
        new_values=json.dumps(redact_sensitive(new_values), ensure_ascii=False, default=str) if new_values else ""
    )
```

### 3.6 注册 Honeypot 过于简单

**风险**：现代自动化工具可轻松识别并绕过 Honeypot。

**涉及文件**：
- `backend/app/routers/auth.py:379-384`

**修复方案**：
1. Honeypot 字段名使用随机化命名（每次请求不同）。
2. 增加时间校验：表单提交太快（< 2 秒）视为机器人。
3. 增加行为指纹：检测鼠标移动事件（前端辅助）。

### 3.7 验证码答案是 SHA256 哈希，可被暴力破解

**风险**：a、b 都是 1-10，答案只有 2-20 共 19 种可能，可预计算所有哈希值。

**涉及文件**：
- `backend/app/auth.py:33-43`

**修复方案**：
1. 扩大数字范围到 1-50，或改为乘法。
2. 增加时间戳 salt 到 hash 计算中。
3. 或者替换为图形验证码（如使用 `captcha` 库生成图片）。

### 3.8 CORS `allow_credentials=True` + 宽泛 origin 风险

**风险**：如果 `CORS_ORIGINS=*`，FastAPI 可能反射 Origin header。

**涉及文件**：
- `backend/app/config.py:20`
- `backend/app/main.py:196-202`

**修复方案**：
1. 在 `get_cors_origins()` 中拒绝包含 `*` 的配置。
2. 生产环境强制要求明确的域名列表。

### 3.9 前端 `userRole` 和 `userId` 缓存在 localStorage

**风险**：虽然后端会再次校验，但攻击者可篡改 localStorage 中的 `sparkbin_role` 为 `"admin"`，导致前端 UI 错误显示管理员菜单。

**涉及文件**：
- `frontend/src/services/api.ts:8-9`

**修复方案**：
1. 页面加载时始终调用 `/auth/me` 获取真实角色。
2. 不要在 localStorage 中缓存 `userRole`，只缓存 token。

## 四、低（LOW）—— 可选修复

### 4.1 SQLite `check_same_thread=False`

**风险**：SQLite 多线程访问可能导致数据损坏。

**涉及文件**：
- `backend/app/database.py:11`

**修复方案**：
1. 生产环境强制使用 PostgreSQL。
2. 在启动时检查：如果 `database_url` 以 sqlite 开头且 `debug=False`，发出严重警告。

### 4.2 `RequestSizeLimitMiddleware` 可被流式请求绕过

**风险**：Chunked transfer encoding 可能绕过 body 大小检查。

**涉及文件**：
- `backend/app/main.py:42-58`

**修复方案**：
1. 已在 `uvicorn.run()` 中设置了 `h11_max_request_size=10MB`，这是底层防护。
2. 在生产环境增加 nginx `client_max_body_size 10m;` 作为最终防线。
3. 可以保留当前中间件作为辅助检查。
