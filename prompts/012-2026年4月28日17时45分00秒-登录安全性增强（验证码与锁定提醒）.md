# 任务 #24：登录安全性增强（验证码与锁定提醒）

## 问题描述
当前登录系统的安全机制有两个可优化点：
1. 没有验证码机制，登录接口可能被暴力爆破（虽然有速率限制，但缺少人机验证层）
2. 用户被速率限制锁定时，前端只显示"尝试次数过多"，没有明确的倒计时和视觉提醒
3. 注册接口已有蜜罐字段，但登录接口缺少类似的轻量级防机器人措施

## 修改文件

### 1. `backend/app/schemas.py`（请求体验证）
**允许修改：**
- `LoginRequest`：新增可选字段 `captcha_answer: Optional[str] = None`

**禁止修改：**
- 其他 schema 的定义

### 2. `backend/app/routers/auth.py`（后端安全加固）
**允许修改：**
- `POST /auth/login`：在速率限制检查之前，添加轻量级数学验证码校验
  - 新增 `GET /auth/captcha` 端点：生成简单数学题（如 `3 + 5 = ?`），返回 `{ question: "3 + 5", answer_hash: "sha256(8)" }`
  - `POST /auth/login` 读取 `request.captcha_answer`
  - 当同一 IP 在 5 分钟内失败次数 >= 2 时，强制要求验证码；否则可选
  - 验证答案时计算 `sha256(captcha_answer.strip())` 与 `answer_hash` 比对
- 速率限制响应头：确保 `Retry-After` 返回剩余秒数（当前已返回 300 秒固定值，优化为动态计算）
- 登录失败响应：添加 `X-Login-Attempts-Remaining` 响应头，提示用户还剩几次机会

**禁止修改：**
- `_MAX_LOGIN_ATTEMPTS`（保持 5 次）
- `_LOGIN_WINDOW_SECONDS`（保持 300 秒）
- `check_rate_limit` 和 `record_rate_limit_failure` 的核心 deque 逻辑（只在其外层包装）
- 注册接口的蜜罐字段逻辑

### 3. `backend/app/auth.py`（验证码工具函数）
**允许修改：**
- 新增 `_captcha_store: dict[str, tuple[str, float]]` — 内存存储 `{ip: (answer, expire_time)}`
- 新增 `generate_captcha(ip: str) -> dict` — 生成 1-20 的加法题，存储答案，5 分钟过期
- 新增 `verify_captcha(ip: str, answer: str) -> bool` — 验证并清除记录

**禁止修改：**
- 不引入外部验证码库（如 PIL/Captcha），保持纯文本数学题
- 不引入 Redis，保持内存存储

### 4. `frontend/src/components/LoginModal.tsx`（前端安全交互）
**允许修改：**
- 登录表单新增验证码输入行：
  - 当后端返回 `require_captcha: true` 或在登录失败 2 次后，显示验证码输入框
  - 验证码问题文本显示在输入框左侧（如 `3 + 5 =`）
  - 添加刷新按钮重新获取验证码
- 锁定状态优化：
  - 当后端返回 429 时，解析 `Retry-After` 响应头，显示倒计时（如"请等待 4 分 32 秒后重试"）
  - 倒计时期间禁用登录按钮和输入框
  - 倒计时结束后自动启用

**禁止修改：**
- 三栏切换结构
- 注册表单的蜜罐字段
- OAuth 按钮区域

### 5. `frontend/src/services/api.ts`（API 适配）
**允许修改：**
- `authApi.login` 支持传入 `captcha_answer?: string`
- 新增 `authApi.getCaptcha()` 调用 `GET /auth/captcha`

**禁止修改：**
- 401/403 处理逻辑
- Token 刷新逻辑

## 测试方案
1. 连续登录失败 2 次 → 第 3 次应显示验证码输入框
2. 输入错误验证码 → 返回「验证码错误」
3. 连续失败 5 次 → 返回 429，前端显示倒计时，按钮禁用
4. 等待倒计时结束 → 按钮恢复，可重新登录
5. 正确验证码 + 正确密码 → 正常登录
6. `npm run build` 无 TypeScript 编译错误
