# Pointer：AI 建议功能完整设计方案

> **功能导向**：不分前端/后端，按功能闭环推进。  
> **目标分支**：`master`（主分支）。  
> **文档先行**：本文档为开发唯一依据，开发前必须通读。

---

## 1. 问题背景

在项目详情页的"想法（Idea）"阶段，面板上有若干便利贴（核心痛点、目标用户、使用场景、解决方案、差异化价值）。当前"AI 建议"按钮点击后：

1. **无反应或报错**：前端直接调用 AI API（不走后端代理），网络/API Key 问题导致失败。错误只在 `console.error` 输出，用户看不到任何反馈。
2. **不填充面板**：AI 返回的是一段纯文本建议，仅显示在面板上方，**不会自动填充到各张便利贴**。
3. **丢失原始想法**：用户在创建项目时输入的初步想法没有被保存。AI 优化后的语言虽然规整，但不能百分之百准确表达用户原意。

---

## 2. 方案概述

### 2.1 目标

点击"AI 建议"后：
1. 走**后端代理**调用 AI（前端不接触 API Key）。
2. AI 返回**结构化数据**，在对话框中预览。
3. 用户可选择"智能合并"或"全部覆盖"，将建议应用到便利贴。
4. 创建项目时保存用户的**原始想法**，与 AI 优化后的内容分离存储。

### 2.2 设计原则

- **复用现有基础设施**：`CreateProjectModal` 的对话框风格、`Toast` 错误提示组件已有，不再新建。
- **用户可控**：AI 建议必须经过用户确认后才生效，不可自动覆盖。
- **原始想法保真**：`original_idea` 字段只保存用户原话，后续不可被 AI 修改。

---

## 3. AI 建议对话框设计

### 3.1 触发流程

```
用户点击"AI 建议"按钮
  → 显示对话框（加载态）
  → 前端调用 POST /ai/idea-suggest
  → 后端调用 AI 代理，返回结构化建议
  → 对话框切换为预览态
  → 用户选择操作
  → 关闭对话框，更新面板
```

### 3.2 对话框 UI 结构

复用 `CreateProjectModal` 的 Brutalist 风格：

- **Header**：`// AI 建议预览` + 关闭按钮
- **Body**：
  - 左侧：当前便利贴（灰色、只读）
  - 右侧：AI 建议（高亮、可逐条编辑）
  - 每条便利贴对比展示：`标题` + `当前内容 → 建议内容`
- **Footer**：
  - `取消` — 不应用任何修改
  - `智能合并` — 只覆盖内容为默认占位符的便利贴
  - `全部覆盖` — 用 AI 建议替换所有便利贴

### 3.3 智能合并规则

```ts
const DEFAULT_PLACEHOLDERS = [
  '描述你想解决的核心问题...',
  '谁会使用这个产品？',
  '用户在什么情况下会用？',
  '简述核心功能...',
  '与现有方案相比，你的优势是什么？',
  '点击编辑...',
];

function isPlaceholder(note: StickyNote): boolean {
  return DEFAULT_PLACEHOLDERS.some(p => note.content.includes(p));
}
```

**智能合并行为**：
- 对 `isPlaceholder === true` 的便利贴：用 AI 建议内容替换。
- 对 `isPlaceholder === false` 的便利贴：保留原内容，但在对话框右侧显示 AI 建议，让用户手动选择是否采纳（通过每行旁边的单选按钮或复选框）。

### 3.4 AI 返回结构

```json
{
  "notes": [
    { "title": "核心痛点", "content": "AI 生成的痛点描述..." },
    { "title": "目标用户", "content": "AI 生成的用户画像..." },
    { "title": "使用场景", "content": "AI 生成的场景..." },
    { "title": "解决方案", "content": "AI 生成的方案..." },
    { "title": "差异化价值", "content": "AI 生成的差异化..." }
  ]
}
```

## 4. 原始想法保存设计

### 4.1 现状

`CreateProjectModal.tsx` 的 Step 1 已让用户输入原始想法（`painPoint` textarea），Step 2 展示"你描述的是："——UI 层面已经展示了原始想法。但 `handleCreate` 只调用了 `createProject(title, painPoint)`，原始输入未被单独保存。

### 4.2 修改点

1. **后端 `Project` 模型**新增字段：
   ```python
   original_idea = Column(Text, default="", nullable=False)
   ```
2. **Schema 同步**：`ProjectCreate`、`ProjectUpdate`、`ProjectInfo`、`ProjectDetail` 都加上 `original_idea: str = ""`。
3. **前端 `CreateProjectModal`**：
   - `handleCreate` 调用时，把 Step 1 中用户输入的原始值传入 `original_idea`。
   - `pain_point` 继续存 AI 优化后的版本（或用户 Step 3 手动修改后的版本）。

### 4.3 数据流

```
Step 1: 用户输入原始想法 → 暂存到 local state（painPoint）
Step 2: AI 分析并生成维度（基于 painPoint）
Step 3: 用户确认/修改后 → painPoint 可能被更新
创建项目时:
  original_idea = Step 1 的原始值（不可变）
  pain_point = Step 3 的最终值（可变）
```

## 5. 错误提示接入

### 5.1 现状

`Toast.tsx` 和 `useToast.ts` 已存在，风格统一（Brutalist、等宽字体、3 秒自动消失）。但 `IdeaStage.tsx` 中使用了 `console.error`，用户看不到。

### 5.2 修改点

在 `IdeaStage.tsx` 中：

```ts
import { useToast } from '../hooks/useToast';

const { showToast } = useToast();

const getAiSuggestion = async () => {
  setIsGenerating(true);
  try {
    const suggestion = await aiService.generateIdeaSuggestion(...);
    setAiSuggestion(suggestion);
  } catch (error) {
    showToast('获取 AI 建议失败，请检查网络或 AI 配置', 'error');
  } finally {
    setIsGenerating(false);
  }
};
```

同理，所有调用 AI 的地方都应接入 `useToast`，取代裸的 `console.error`。

## 6. 后端 API 设计

### 6.1 新增端点：POST /ai/idea-suggest

**请求体**：
```json
{
  "project_id": "uuid",
  "title": "项目标题",
  "pain_point": "痛点描述",
  "original_idea": "用户原始想法",
  "current_notes": [
    { "title": "核心痛点", "content": "当前内容" },
    { "title": "目标用户", "content": "当前内容" }
  ]
}
```

**响应体**：
```json
{
  "notes": [
    { "title": "核心痛点", "content": "AI 建议内容" },
    { "title": "目标用户", "content": "AI 建议内容" }
  ]
}
```

**实现位置**：`backend/app/routers/ai.py`

**AI Prompt 设计**：
```
你是一个资深产品顾问，擅长帮助创业者完善产品想法。

用户原始想法：{original_idea}
当前痛点描述：{pain_point}
项目标题：{title}

当前便利贴内容：
- 核心痛点: {content}
- 目标用户: {content}
...

请根据用户的原始想法，优化并填充以下 5 个维度的内容。必须返回 JSON 格式：
{
  "notes": [
    {"title": "核心痛点", "content": "..."},
    {"title": "目标用户", "content": "..."},
    {"title": "使用场景", "content": "..."},
    {"title": "解决方案", "content": "..."},
    {"title": "差异化价值", "content": "..."}
  ]
}

要求：
1. 忠实于用户原始意图，不要过度发挥
2. 每个维度 1-2 句话，简洁有力
3. 内容 actionable，符合 Vibe 开发理念
```

### 6.2 后端 AI 代理服务

在 `backend/app/services/ai_proxy.py` 中新增方法：

```python
async def generate_idea_suggestions(
    self,
    title: str,
    pain_point: str,
    original_idea: str,
    current_notes: list[dict],
    provider: AIProvider,
    model: str,
) -> dict:
    ...
```

### 6.3 Project 模型与路由修改

- `backend/app/models.py`：`Project` 表新增 `original_idea` 字段。
- `backend/app/schemas.py`：相关 schema 新增 `original_idea: str = ""`。
- `backend/app/routers/projects.py`：创建/更新项目时处理 `original_idea`。
- `backend/app/main.py`：SQLite 兼容性补丁中动态添加 `original_idea` 列。

## 9. 测试计划

### 9.1 单元测试

1. **后端**：
   - `POST /ai/idea-suggest` 返回正确 JSON 结构
   - `Project` 模型能正确序列化/反序列化 `original_idea`
   - 创建项目时 `original_idea` 被正确保存
2. **前端**：
   - `isPlaceholder` 函数正确识别默认占位符
   - 智能合并逻辑正确（保留用户修改，覆盖占位符）

### 9.2 集成测试

1. 创建项目 → 输入原始想法 → 保存后查看 `GET /projects/{id}` 返回包含 `original_idea`
2. 进入 Idea 阶段 → 点击"AI 建议" → 对话框弹出 → 选择"全部覆盖" → 便利贴内容更新
3. 断网状态下点击"AI 建议" → Toast 显示错误信息

### 9.3 闭合循环（必须执行）

1. `cd backend && python -c "import app.main; print('import ok')"`
2. `cd backend && python start.py` 启动后端
3. `cd frontend && npm run build` 确认 TypeScript 无错误
4. 浏览器端到端验证：创建项目 → AI 建议 → 合并 → 刷新页面数据持久化
