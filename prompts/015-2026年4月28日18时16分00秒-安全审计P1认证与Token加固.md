# 任务 #26：安全审计 P1 认证与 Token 生命周期加固

## 问题描述

经安全审计，发现 8 个 P1 级中危漏洞，集中在认证生命周期管理、OAuth 安全和环境配置校验。这些问题不会立即导致数据泄露，但会显著削弱系统的安全防御能力，需要在本周内修复。

### P1-1：修改密码 / 重置密码后未使现有 Token 失效
- **位置**：`backend/app/routers/auth.py:156-205` (change-password)、`backend/app/routers/auth.py:473-506` (reset-password)
- **根因**：用户修改密码或重置密码后，之前签发的 access token 和 refresh token 在过期前（最长 7 天）仍然完全有效。如果 token 已泄露，改密码无法止损。
- **修复方案**：引入 **Token Version** 机制。在 `User` 模型增加 `token_version` 整数字段，JWT payload 中包含当前版本号。`get_current_user` 校验 token 中的版本号是否与数据库一致。改密码/重置密码时递增版本号，使所有旧 token 失效。

### P1-2：登出端点未使 Token 失效
- **位置**：`backend/app/routers/auth.py:138-153`
- **根因**：`/auth/logout` 仅记录审计日志，返回成功消息，但 token 在过期前仍可用。
- **修复方案**：结合 Token Version 机制，登出时递增 `token_version`。

### P1-3：Refresh Token 无轮换机制
- **位置**：`backend/app/routers/auth.py:104-135`
- **根因**：使用 refresh token 换取新 access token 时，refresh token 本身不会被更换。如果泄露，攻击者可在 7 天内无限次获取新 access token。
- **修复方案**：实现 Refresh Token Rotation。每次刷新时签发新的 refresh token，并返回给客户端。前端 `api.ts` 的 `refreshAccessToken()` 需要保存新的 refresh token。

### P1-4：OAuth State 参数可重放攻击
- **位置**：`backend/app/routers/auth.py:518-523`
- **根因**：`_create_oauth_state` 使用 JWT 生成 state，有效期 10 分钟，但没有记录已使用的 state。攻击者可在 10 分钟内重放截获的 state。
- **修复方案**：将使用过的 state JWT 的 `jti`（JWT ID）存入短期内存集合 `_used_oauth_states`，验证后立即标记为已使用。`_create_oauth_state` 需要生成包含 `jti` 的 token。

### P1-5：注册速率限制逻辑错误 — 合法请求被计为失败
- **位置**：`backend/app/routers/auth.py:362-366`
- **根因**：
  ```python
  check_rate_limit(req, "注册")
  record_rate_limit_failure(req, "注册")  # 无条件记录失败！
  ```
  每次注册请求（包括成功注册）都被记录为一次失败。后续如果因用户名已存在等原因失败，还会再次记录。
- **修复方案**：删除无条件记录，仅在注册真正失败（honeypot、用户名重复、邮箱重复、密码复杂度不足）时才调用 `record_rate_limit_failure`。

### P1-6：速率限制基于原始连接 IP，代理环境下失效
- **位置**：`backend/app/auth.py:35`
- **根因**：`client_ip = request.client.host` 在 Nginx/Cloudflare/负载均衡后获取到的是代理 IP，导致所有用户共享同一配额，且无法基于真实 IP 限流。
- **修复方案**：新增 `_get_client_ip` 函数，优先从 `X-Forwarded-For` 或 `X-Real-IP` 头获取（仅当信任代理时启用）。

### P1-7：OAuth Token Exchange 未设置超时 — 慢速攻击风险
- **位置**：`backend/app/routers/auth.py:607-616` (Google)、`auth.py:726-735` (GitHub)、`auth.py:919-928` (GitHub connect)
- **根因**：`httpx.post(...)` 没有设置 `timeout`，默认可能等待很长时间。攻击者可构造大量 OAuth 回调请求挂起工作线程。
- **修复方案**：所有 OAuth token exchange 调用显式设置 `timeout=10.0`。

### P1-8：环境变量空字符串可绕过默认密钥安全检查
- **位置**：`backend/app/config.py:58-67`
- **根因**：`get_settings()` 检查 `secret_key == _DEFAULT_SECRET_KEY`，但如果环境变量设置 `SECRET_KEY=`（空字符串），pydantic v2 会设为 `""` 而非默认值，导致检查被绕过。应用可能使用空字符串作为 JWT 签名密钥。
- **修复方案**：增加非空和最小长度检查（至少 32 字符）。

            detail="Invalid authentication credentials",

def _get_client_ip(request: Request) -> str:
    forwarded_for = request.headers.get("x-forwarded-for")
    if forwarded_for:
        return forwarded_for.split(",")[0].strip()
    real_ip = request.headers.get("x-real-ip")
    if real_ip:
        return real_ip.strip()
    return request.client.host if request.client else "unknown"


def check_rate_limit(request: Request, action: str) -> None:
    if _is_rate_limit_disabled():

    client_ip = _get_client_ip(request)
    key = f"{client_ip}:{action}"
    ...


def record_rate_limit_failure(request: Request, action: str) -> None:

        return
    client_ip = _get_client_ip(request)
    key = f"{client_ip}:{action}"
    ...

@router.post("/refresh", response_model=TokenPairResponse)
def refresh_token(request: RefreshTokenRequest, db: Session = Depends(get_db)):
    payload = decode_token(request.refresh_token, expected_type="refresh")
    if payload is None:
        raise HTTPException(status_code=401, detail="Refresh token 无效或已过期")

    username: str = payload.get("sub")
    if username is None:
        raise HTTPException(status_code=401, detail="Refresh token 无效")

    user = db.query(User).filter(User.username == username).first()
    if user is None:
        raise HTTPException(status_code=401, detail="用户不存在")

    # 校验 refresh token 的 version
    token_ver = payload.get("ver", 0)
    if token_ver != user.token_version:
        raise HTTPException(status_code=401, detail="Token has been revoked")

    token_data = {"sub": user.username, "role": user.role.value}
    new_access_token = create_access_token(data=token_data, token_version=user.token_version)
    new_refresh_token = create_refresh_token(data=token_data, token_version=user.token_version)

    return TokenPairResponse(
        access_token=new_access_token,
        refresh_token=new_refresh_token,
    )
```

**4.5 OAuth State 一次性校验：**

import secrets

_used_oauth_states: set[str] = set()

def _create_oauth_state() -> str:
    return create_access_token(
        data={"oauth": True, "jti": secrets.token_urlsafe(16)},
        expires_delta=timedelta(minutes=10)
    )

def _verify_oauth_state(state: str) -> bool:
    payload = decode_token(state, expected_type="access")
    if payload is None or payload.get("oauth") is not True:
        return False
    jti = payload.get("jti")
    if not jti or jti in _used_oauth_states:
        return False
    _used_oauth_states.add(jti)
    return True
```

## 测试方案

### 后端测试

1. **Token Version 机制测试**：
   - 用户 A 登录，获取 token_pair
   - 用户 A 调用 `/auth/change-password` 修改密码
   - 使用旧 access_token 请求 `/auth/me` → 期望 401 "Token has been revoked"
   - 使用旧 refresh_token 请求 `/auth/refresh` → 期望 401
   - 重新登录，获取新 token → 期望正常

2. **登出使 Token 失效测试**：
   - 用户登录，获取 token
   - 调用 `/auth/logout`
   - 使用原 token 请求 `/auth/me` → 期望 401

3. **Refresh Token Rotation 测试**：
   - 用户登录，记录 refresh_token_1
   - 调用 `/auth/refresh` 传入 refresh_token_1 → 期望返回新的 access_token + refresh_token_2
   - 再次用 refresh_token_1 调用 `/auth/refresh` → 期望 401（因为 token_version 已递增... 等等，这里需要重新考虑）

   **重要注意**：当前 Refresh Token Rotation 方案与 Token Version 机制存在冲突！因为 `refresh_token` 递增 `token_version` 会导致所有 token 失效，包括当前有效的 access token。正确的 Refresh Token Rotation 不应递增 `token_version`，而应使用 token 黑名单或 `jti` 机制。

   **修正后的 Refresh Token Rotation 方案**：
   - 不在 refresh 时递增 `token_version`
   - 而是在 refresh 时返回新的 token pair（access + refresh），旧 refresh token 自然废弃（客户端不再持有）
   - 如果攻击者偷了旧 refresh token 再次使用，由于客户端已更新，攻击者会拿到新 token，但客户端下次刷新会失败，从而触发重新登录
   - 更安全的做法：在 `decode_token` 时检查 `jti` 是否在黑名单中（需要 Redis/内存存储）

   **鉴于复杂度，P1-3 可以简化为**：refresh 时返回新的双 token，前端保存。不在后端维护黑名单（留待 P0 之后的迭代优化）。

   因此 `refresh_token` 端点**不递增** `token_version`，只签发新的 token pair：
   ```python
   new_access_token = create_access_token(data=token_data, token_version=user.token_version)
   new_refresh_token = create_refresh_token(data=token_data, token_version=user.token_version)
   ```

4. **OAuth State 重放测试**：
   - 正常完成一次 Google OAuth 登录
   - 在 10 分钟内，用同一个 `state` 参数再次请求 `/auth/oauth/google/callback` → 期望 400 "Invalid OAuth state"

5. **注册速率限制修复验证**：
   - 成功注册一个新用户 → 检查该 IP 的注册失败计数应为 0
   - 用同一 IP 连续 5 次提交已存在的用户名 → 第 6 次应被 429 锁定

6. **真实 IP 测试**：
   - 后端启动后，模拟带 `X-Forwarded-For: 1.2.3.4` 的请求登录失败 → 审计日志中 `ip_address` 应记录为 `1.2.3.4`

7. **OAuth 超时测试**：
   - 通过代码审查确认所有 `httpx.post` 在 OAuth 流程中均有 `timeout=10.0`

8. **环境变量空字符串测试**：

## 关键注意事项

1. **Token Version 与现有用户的兼容性**：由于 `token_version` 新增列默认值为 0，现有用户的 token 中如果没有 `ver` claim，`payload.get("ver", 0)` 会返回 0，与数据库默认值一致，所以不会破坏现有会话。
2. **Refresh Token Rotation 不触发版本递增**：刷新 token 时不应递增 `token_version`，否则会导致当前有效的 access token 也失效。refresh 只签发新的 token pair，由客户端替换旧值。
3. **OAuth State 集合的内存管理**：`_used_oauth_states` 是内存集合，应用重启后清空。这是可接受的（state 有效期仅 10 分钟）。如需更持久化，可考虑使用 TTL 缓存。
4. **`change_password` 递增 `token_version` 后，当前请求不会立即失效**：因为 `change_password` 端点本身已经通过 `get_current_user` 完成了校验，修改密码的响应仍然能正常返回。但后续请求使用该 token 会被拒绝。
