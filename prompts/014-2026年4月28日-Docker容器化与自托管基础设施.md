# 任务 #14：Docker 容器化与自托管基础设施

## 目标
为 SparkBin 构建完整的容器化基础架构，将自托管门槛从"半天手动配置"降到"5 分钟一键启动"。

## 现状
- 前端：React 18 + Vite + TypeScript，构建后是纯静态文件
- 后端：FastAPI + SQLAlchemy + PostgreSQL，Python 3.11+
- 现有 `SELF_HOSTING.md` 只有手动部署指南，明确写了"Docker (Community Contribution Welcome)"
- 无 `.env.example`，无容器编排文件

## 执行步骤

### 1. 创建 `docker/Dockerfile.backend`
- 基于 `python:3.11-slim`
- 工作目录 `/app`
- 先复制 `requirements.txt` 安装依赖（利用 Docker 缓存层）
- 再复制 `backend/` 全部代码
- 暴露端口 `8000`
- 启动命令：`uvicorn app.main:app --host 0.0.0.0 --port 8000`
- **关键**：启动时自动运行 Alembic migrate（`alembic upgrade head` 或兼容 SQLite 的初始化逻辑）

### 2. 创建 `docker/Dockerfile.frontend`
- 多阶段构建
- 阶段一（build）：`node:20-alpine`，安装依赖、运行 `npm run build`
- 阶段二（serve）：`nginx:alpine`，复制构建产物到 `/usr/share/nginx/html`
- 提供自定义 `nginx.conf`：
  - 监听 `80`
  - 所有路由回退到 `index.html`（支持 React Router）
  - 反向代理 `/api` 到后端服务（通过环境变量 `VITE_API_URL` 构建时注入）

### 3. 创建 `docker/nginx.conf`
- 静态文件服务
- SPA fallback：`try_files $uri $uri/ /index.html`
- `/api` 反向代理到 `http://backend:8000`

### 4. 创建根目录 `.env.example`
包含所有必要配置项，每个都有注释说明：
```bash
# 数据库
DATABASE_URL=postgresql://sparkbin:changeme@postgres:5432/sparkbin

# AI（管理员在后台配置，此处为后端加密密钥）
SECRET_KEY=change-me-in-production-min-32-chars-long!!

# 前端构建时注入
VITE_API_URL=http://localhost:8000

# OAuth（可选）
GOOGLE_CLIENT_ID=
GOOGLE_CLIENT_SECRET=
GITHUB_CLIENT_ID=
GITHUB_CLIENT_SECRET=

# 邮件（可选，用于密码重置）
SMTP_HOST=
SMTP_PORT=587
SMTP_USER=
SMTP_PASS=
```

### 5. 创建 `docker-compose.yml`（根目录）
服务编排：
- `postgres`：官方 `postgres:15-alpine`，持久化卷 `postgres_data`
- `backend`：依赖 `postgres`，环境变量从 `.env` 读取
- `frontend`：依赖 `backend`，Nginx 服务
- `caddy`（可选，注释状态）：自动 HTTPS 反向代理，适合有域名的用户

### 6. 创建 `docker-compose.production.yml`
- 引用预构建镜像（`ghcr.io/yourname/sparkbin-backend:latest`）而非本地 build
- 包含 Caddy/Traefik 自动 HTTPS 配置模板

### 7. 更新 `SELF_HOSTING.md`
保留现有手动部署章节作为"高级选项"，新增：
- "Docker 快速开始"（置顶）
- 步骤：克隆仓库 → `cp .env.example .env` → `docker compose up -d`
- 默认访问地址：`http://localhost`
- 初始账号：`admin` / `changeme`（首次登录后必须修改）
- 升级步骤：`docker compose pull && docker compose up -d`
- 备份策略：`docker exec postgres pg_dump ...`

### 8. 创建 `.dockerignore`
- 排除 `node_modules/`、`__pycache__/`、`.git/`、`*.db`（本地 SQLite）

## 约束条件

**允许修改/新建：**
- `docker/Dockerfile.backend`（新建）
- `docker/Dockerfile.frontend`（新建）
- `docker/nginx.conf`（新建）
- `.env.example`（新建）
- `docker-compose.yml`（新建）
- `docker-compose.production.yml`（新建）
- `.dockerignore`（新建）
- `SELF_HOSTING.md`（修改）

**禁止修改：**
- `backend/alembic/versions/` 下任何已有迁移文件
- 前端/后端业务代码（此任务只做容器化，不改功能）
- `frontend/vite.config.ts` 中的 `server.proxy`（Nginx 层面处理反向代理）

## 测试方案
1. `docker compose build` 成功（前后端镜像都能构建）
2. `docker compose up -d` 后：
   - `http://localhost` 能访问前端页面
   - `http://localhost/api/health` 或任意后端端点返回 200
   - 能完成注册/登录流程
   - 创建项目后刷新页面数据持久化（PostgreSQL 工作正常）
3. `docker compose down && docker compose up -d` 后数据不丢失（postgres_data 卷持久化）
4. `backend/tests/` 中的测试能在容器内运行（可选验证）

## 提交标准
- 本地 `docker compose up` 端到端验证通过
- 无敏感信息硬编码在 Dockerfile 中（全部走 `.env`）
- `README.md` 中新增 Docker 部署徽章/链接
