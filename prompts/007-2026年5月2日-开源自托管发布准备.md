# Prompt: SparkBin 开源自托管发布准备

## 背景

SparkBin 是一个面向独立开发者和 vibe coder 的 AI-native 项目教练工具，技术栈为 FastAPI + React + Vite + Tailwind。项目已完成核心功能开发，具备 Docker 容器化能力。现需完成从私有开发到 GitHub 开源发布的全部准备工作，确保任何开发者都能通过一行命令 `docker compose up -d` 完成自托管部署。

## 目标

将 SparkBin 代码库从当前状态推进到可直接在 GitHub 上公开发布并供全球开发者自托管使用的成熟开源项目状态。

## 当前已完成的基础工作

- 许可证已从 Elastic License 2.0 更换为 **AGPL-3.0**（`LICENSE`）
- Docker 文件已就绪：`docker/Dockerfile.backend`（非 root + HEALTHCHECK）、`docker/Dockerfile.frontend`（多阶段 Nginx 构建）、`docker/nginx.conf`（SPA fallback + API proxy）
- Docker Compose 已就绪：`docker-compose.yml`（开发：PostgreSQL + backend + frontend + 可选 Ollama）、`docker-compose.production.yml`（生产：Caddy HTTPS + 预构建镜像）
- 自托管文档已就绪：`SELF_HOSTING.md`（快速开始、手动部署、生产部署、Ollama 离线 AI、故障排除）
- README 已更新：包含 Docker Quick Start、Features、Tech Stack、Screenshots 占位符
- 根目录 `.env.example` 已补全所有关键环境变量（含自托管默认值 `ENABLE_PAYMENTS=false`、`ENABLE_SAAS_FEATURES=false`）

## 需要执行的详细任务

### 任务一：仓库安全与清理
1. 扫描 Git 历史，确认无硬编码密钥（Stripe live key、Resend API key、真实密码）残留
2. 清理不应进入公开仓库的文件：
   - `frontend/backend_run.err`、`frontend_run.err`、`backend_run.out` 等日志文件（加入 `.gitignore` 并删除已跟踪文件）
   - `prompts/` 目录中的旧历史提示词（评估保留价值，建议保留以展示开发历程，或移入 `docs/prompts-history/`）
3. 检查 `.gitignore` 完整性，确保 `.env`、`*.db`、`*.log`、`node_modules`、`.pyc` 不会被提交

### 任务二：GitHub 仓库基础设施
1. 在 `.github/` 目录下创建：
   - `workflows/backend-tests.yml` — Python 3.11 + pytest CI
   - `workflows/frontend-build.yml` — Node 20 + build CI
   - `workflows/docker-publish.yml` — Release 时自动构建并推送 `ghcr.io/<owner>/sparkbin-backend` 和 `sparkbin-frontend`
   - `dependabot.yml` — Python pip + npm 依赖自动安全检查
   - `ISSUE_TEMPLATE/bug_report.yml` 和 `feature_request.yml` — 标准化 Issue 模板
   - `PULL_REQUEST_TEMPLATE.md` — PR 检查清单
2. 创建 `CODE_OF_CONDUCT.md`（Contributor Covenant 标准模板）
3. 规范化 `CHANGELOG.md` 为 [Keep a Changelog](https://keepachangelog.com/) 格式

### 任务三：文档最终完善
1. 更新 `README.md`：
   - 替换 Screenshots 占位符为真实运行截图（各阶段界面：Idea、Validate、Prototype、Ship、Grow、Monetize）
   - 添加最低系统要求（CPU / 内存 / 磁盘）
   - 更新 License badge 为 AGPL-3.0（已更新，确认显示正确）
2. 更新 `SELF_HOSTING.md`：
   - 补充完整环境变量表格（所有变量、是否必填、默认值、用途说明）
   - 补充从 Elastic License 迁移到 AGPL-3.0 的简短说明
3. 可选：在关键源文件顶部添加简短 LICENSE 头部注释（用脚本批量添加）

### 任务四：Docker 与部署验证
1. 在全新环境（或干净的虚拟机/Docker Desktop）中执行完整部署测试：
   ```bash
   git clone <repo-url>
   cd sparkbin
   cp .env.example .env
   # 编辑 SECRET_KEY, ENCRYPTION_KEY, DEFAULT_PASSWORD
   docker compose up -d
   # 访问 http://localhost
   # 登录 admin / <DEFAULT_PASSWORD>
   ```
2. 验证离线模式：不配置任何外部 API key，确认核心功能（项目管理、阶段编辑、看板）正常运行
3. 验证 Ollama 本地 AI：取消注释 docker-compose.yml 中的 ollama 服务，pull llama3.2，在 Admin 面板启用后测试 AI 建议
4. 确认生产 compose 文件中的镜像名 `ghcr.io/yourname/...` 已替换为实际 GitHub 用户名

### 任务五：发布执行
1. 创建 Git Tag `v1.0.0` 并推送
2. 在 GitHub 上创建 Release，撰写 Release Notes（包含 Docker 部署命令、最低配置、已知问题）
3. 配置 GitHub Container Registry 权限，确保 Actions 能推送镜像
4. 设置仓库 Topics：`project-management`, `ai`, `self-hosted`, `fastapi`, `react`, `indie-hacker`
5. 开启 Discussions 和 Security Advisories

## 技术约束与规范

- **绝对禁止停止服务**：执行测试时必须在不中断当前开发服务器的前提下进行，使用独立端口或 Docker 网络隔离
- **禁止修改 migrations 文件夹下的任何文件**
- **禁止在代码中硬编码任何密钥或密码**，所有配置必须通过 `.env` 注入
- **使用命名导出**，不引入新的默认导出风格
- **不要精简输出**：提供详尽的解释、代码注释和多种方案对比
- **完整代码输出**：永远不要使用 "..." 省略代码，必须输出完整可运行代码

## 验收标准

- [ ] `docker compose up -d` 在全新机器上 5 分钟内启动成功
- [ ] 不配置任何外部服务时，核心功能 100% 可用
- [ ] GitHub Actions 所有 CI 流水线通过（backend tests、frontend build）
- [ ] Release 页面包含可直接复制运行的 Docker Compose 命令
- [ ] 无敏感信息泄露（通过 `git log -p -S 'sk_'` 等命令验证）
- [ ] 所有文档（README、SELF_HOSTING、CONTRIBUTING、SECURITY、CHANGELOG）无死链、无过期信息

## 输出格式

每完成一个任务，更新 `OPEN_SOURCE_CHECKLIST.md` 中对应的复选框状态，并在最终回复中提供：
1. 已执行操作的摘要列表
2. 仍需用户手动决策的事项（如：是否保留 prompts/ 历史文件、GitHub 用户名替换、截图拍摄）
3. 发布前最后 3 步检查清单
